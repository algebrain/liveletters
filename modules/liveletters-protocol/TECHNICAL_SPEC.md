# liveletters-protocol

## Назначение

`liveletters-protocol` это библиотека межклиентского протокола LiveLetters. Она задает структуру протокольных сообщений, технический payload, MIME mapping и версионирование форматов.

## Зона ответственности

- структура envelope;
- набор технических заголовков;
- схема payload;
- сериализация и десериализация;
- MIME layout;
- schema versioning;
- backward compatibility policy;
- формат технических ошибок разбора.

## Что модуль не должен делать

- содержать IMAP/SMTP код;
- применять доменные события к состоянию;
- работать с SQLite;
- знать о frontend rendering;
- принимать решения о правах доступа.

## Основные сущности протокола

- `ProtocolMessage`;
- `MessageEnvelope`;
- `TechnicalPayload`;
- `HumanReadableBody`;
- `HeaderSet`;
- `PayloadVersion`.

## Обязательные требования

- каждое доставляемое протокольное сообщение содержит человекочитаемую часть;
- каждое доставляемое протокольное сообщение содержит технический payload;
- формат должен быть версионируемым;
- парсинг должен быть устойчив к частично поврежденным сообщениям;
- формат ошибок должен позволять диагностику.

## Требования к API

- функции encode/decode;
- валидация схемы;
- отдельные типы ошибок;
- поддержка совместимости версий;
- явное разделение между envelope и domain payload.

## Требования к структуре каталога

- `src/envelope`;
- `src/headers`;
- `src/mime`;
- `src/payload`;
- `src/versioning`;
- `tests/fixtures`;
- `tests/compatibility`.

## Требования к тестам

Покрытие тестами обязательно.

Минимум:

- tests на round-trip encode/decode;
- tests на MIME representation;
- tests на compatibility между версиями схем;
- tests на malformed payload;
- tests на отсутствие обязательных полей;
- snapshot или fixture-based tests для типовых сообщений.

## Требования к документации

Обязательна документация:

- спецификация структуры сообщения;
- правила версионирования;
- список заголовков;
- примеры типовых сообщений;
- правила обработки ошибок и несовместимости версий.

## Критерии готовности

- библиотека умеет собирать и разбирать протокольные сообщения;
- схема документирована;
- совместимость версий тестируется;
- boundary между domain payload и email envelope явен.
