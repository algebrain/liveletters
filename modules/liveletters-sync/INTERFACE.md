# `liveletters-sync` INTERFACE

## Назначение

`liveletters-sync` это внешний программный интерфейс контура синхронизации и применения входящих сообщений LiveLetters.

Если смотреть на уже пройденные модули по порядку:

- `liveletters-domain` задает форму предметных фактов;
- `liveletters-protocol` задает форму технического сообщения;
- `liveletters-store` хранит materialized state и технические журналы;
- `liveletters-app-core` собирает прикладные сценарии исходящего контура;
- `liveletters-mail` умеет доставать технический payload из email;
- `liveletters-sync` отвечает за противоположный поток: как входящее письмо превращается в примененное или отклоненное локальное изменение.

Именно здесь находится логика, которая отвечает на такие вопросы:

- удалось ли распознать письмо как LiveLetters-сообщение;
- удалось ли декодировать payload;
- является ли событие дубликатом;
- это новое изменение или replay;
- хватает ли зависимостей, чтобы применить событие сейчас;
- нужно ли отложить событие;
- удалось ли применить его к локальному состоянию.

Этот модуль уже ближе к прикладной логике, чем `mail`, но все еще не является UI boundary и не является полным runtime-приложением.

## Где находится интерфейс

- crate: `liveletters-sync`
- точка подключения: `src/lib.rs`

Наружу модуль экспортирует:

- `SyncEngine`
- `SyncReport`
- `SyncMessageOutcome`
- `SyncError`

## Что считается внешним интерфейсом этого модуля

С практической точки зрения внешний интерфейс `liveletters-sync` это:

1. `SyncEngine::new(store)`;
2. метод `ingest_batch(...)`;
3. метод `reprocess_deferred()`;
4. `SyncReport` как итог одной sync-операции;
5. `SyncMessageOutcome` как словарь возможных исходов;
6. `SyncError` как единый внешний тип ошибок sync-контура.

Именно этим интерфейсом пользуются:

- `liveletters-app-core`, когда запускает повторную обработку deferred events;
- тесты sync-пайплайна;
- потенциально backend orchestration layer, если ему нужно запускать ingest batch.

## Главный объект: `SyncEngine`

### Зачем нужен `SyncEngine`

`SyncEngine` это фасад sync-контура.

Он нужен затем, чтобы верхний код не собирал вручную длинную цепочку:

- разобрать email;
- извлечь части;
- декодировать protocol message;
- проверить duplicate;
- сохранить raw message;
- сохранить raw event;
- применить payload;
- решить, это applied, invalid, replay, deferred или unauthorized.

Идея интерфейса здесь такая:

- верхний слой передает входящие письма;
- `SyncEngine` сам проходит по pipeline;
- на выходе возвращает структурированный отчет.

### Как создается `SyncEngine`

`SyncEngine::new(store: &Store) -> SyncEngine`

Это означает:

- движок синхронизации работает поверх уже открытого `Store`;
- жизненный цикл хранилища контролируется внешним слоем;
- sync не открывает и не закрывает базу сам.

Такой контракт удобен:

- для тестов;
- для `app-core`;
- для backend integration слоя.

## Основной сценарий: `ingest_batch(...)`

### Что принимает

`ingest_batch(messages: Vec<ReceivedEmail>) -> Result<SyncReport, SyncError>`

На вход подается batch уже полученных транспортных писем.

Важно понимать границу:

- `SyncEngine` не получает сокет, mailbox или SMTP/IMAP config;
- он получает уже готовые `ReceivedEmail`.

То есть transport-слой должен сначала сделать свою часть работы, и только потом управление переходит в sync.

### Что реально делает `ingest_batch`

Для каждого сообщения batch-пайплайн выглядит так:

1. пытается распарсить raw email;
2. пытается извлечь человекочитаемую и техническую части;
3. пытается декодировать `ProtocolMessage`;
4. валидирует согласованность envelope и payload;
5. проверяет, не видели ли это событие раньше;
6. сохраняет технические следы в store;
7. пытается применить payload к локальному состоянию;
8. если сразу применить нельзя, решает, откладывать ли событие;
9. формирует outcome по каждому письму.

На выходе по всему batch возвращается:

- `SyncReport`

То есть sync-слой не прячет результат своей работы за простым `Ok(())`. Он всегда пытается объяснить, что именно произошло с каждым сообщением.

## Второй сценарий: `reprocess_deferred()`

### Зачем он нужен

Не все события можно применить сразу.

Типичный пример:

- комментарий пришел раньше поста;
- событие формально корректное;
- но локально еще не хватает нужного состояния.

В таких случаях sync-контур не должен просто выбрасывать событие. Для этого и существует deferred queue.

### Что делает `reprocess_deferred()`

`reprocess_deferred() -> Result<SyncReport, SyncError>`

Этот метод:

1. читает все `DeferredEventRecord` из store;
2. заново пытается применить их payload;
3. если получилось, удаляет их из deferred queue;
4. если по-прежнему нельзя применить, оставляет их отложенными;
5. если выясняется replay, invalid или unauthorized, фиксирует уже окончательный outcome;
6. возвращает `SyncReport`.

То есть это не “перезапусти всё заново”, а именно отдельный интерфейс повторной обработки ранее отложенных событий.

## Почему `SyncReport` важен

`SyncReport` это не служебная деталь, а важная часть внешнего интерфейса.

Смысл этого типа:

- дать верхнему слою единый результат sync-операции;
- не заставлять его самому догадываться, что произошло внутри;
- дать основу для diagnostics, counters и дальнейшего orchestration.

### Что хранит `SyncReport`

Он хранит список outcomes:

- по одному на каждое обработанное сообщение или deferred event.

Создается через:

- `SyncReport::new(outcomes)`

Читается через:

- `outcomes()`

То есть верхний слой получает не только факт успеха, а детальную картину результатов.

## `SyncMessageOutcome`: зачем нужен этот тип

`SyncMessageOutcome` это словарь возможных исходов обработки сообщения.

Это один из самых важных интерфейсов sync-модуля, потому что именно он выражает внешне наблюдаемую семантику sync-пайплайна.

Сейчас наружу экспортируются такие варианты:

- `Applied`
- `Duplicate`
- `Replay`
- `Unauthorized`
- `Invalid`
- `Deferred`
- `Malformed`

## Что означает каждый outcome

### `Applied { message_id, event_id }`

Это лучший нормальный исход.

Он означает:

- письмо удалось разобрать;
- payload оказался валиден;
- событие не является duplicate/replay;
- его удалось применить к локальному состоянию.

Для верхнего слоя это сигнал: сообщение успешно вошло в локальную реплику.

### `Duplicate { message_id, event_id }`

Это означает:

- событие уже встречалось раньше с тем же `event_id`;
- повторное применение не требуется.

Это важный outcome, потому что email-транспорт по своей природе допускает повторную доставку.

С точки зрения интерфейса это не ошибка инфраструктуры и не сбой sync-слоя, а нормальный класс исхода.

### `Replay { message_id, event_id, reason }`

Это означает:

- событие пришло с новым `event_id` или в новом контексте;
- но фактически пытается повторно материализовать уже достигнутое состояние;
- повторное применение не должно менять local state.

То есть replay отличается от duplicate:

- duplicate это “видели именно это событие”;
- replay это “такой переход состояния уже фактически произошел раньше”.

Это важное различие для диагностики и понимания поведения системы.

### `Unauthorized { message_id, event_id, reason }`

Это означает:

- сообщение структурно корректно;
- payload распознан;
- но отправитель или тип действия не прошел текущую policy авторизации.

То есть это не malformed письмо и не технический сбой, а корректно сформированное, но запрещенное действие.

### `Invalid { message_id, event_id, reason }`

Это означает:

- письмо и payload удалось разобрать;
- но с точки зрения согласованности данных событие противоречиво или некорректно.

Например:

- mismatch между `envelope` и `payload`;
- или другая проблема структурной консистентности уже после декодирования.

То есть `Invalid` отличается от `Malformed`:

- malformed нельзя даже нормально разобрать;
- invalid можно разобрать, но нельзя считать допустимым событием.

### `Deferred { message_id, event_id, reason }`

Это означает:

- событие не является окончательно отклоненным;
- но прямо сейчас его нельзя применить;
- его нужно сохранить и попробовать позже.

Это важный outcome, потому что LiveLetters сознательно проектируется как система, устойчивая к out-of-order доставке.

### `Malformed { message_id, reason }`

Это означает:

- письмо не удалось корректно распознать как valid LiveLetters message;
- проблема возникла еще до стадии нормального payload-применения.

Это может быть:

- сломанный email format;
- отсутствие ожидаемой MIME-части;
- невалидный технический JSON.

Для внешнего слоя это сигнал, что сбой произошел на границе transport/protocol parsing.

## Что делает sync-модуль со store

Хотя `liveletters-sync` сам не является persistence-слоем, он активно использует `Store` как технический журнал и materialized state boundary.

Во время работы он:

- записывает `RawMessageRecord`;
- записывает `RawEventRecord`;
- проверяет duplicate через `has_raw_event(...)`;
- записывает `DeferredEventRecord`, если событие откладывается;
- удаляет deferred record после успешной или окончательной переобработки;
- обновляет materialized post/comment state, если событие применено.

То есть внешний интерфейс `SyncEngine` короткий, но за ним стоит вполне богатый persistence-side effect contour.

## Где проходит граница между `mail`, `protocol` и `sync`

Это важно для правильного понимания интерфейса.

### `mail`

Отвечает за:

- получение/отправку email;
- разбор письма;
- извлечение multipart частей.

### `protocol`

Отвечает за:

- структуру технического сообщения;
- кодирование и декодирование JSON payload.

### `sync`

Отвечает за:

- осмысленную обработку входящего сообщения как доменного изменения;
- запись технических журналов;
- duplicate/replay/deferred/unauthorized/invalid semantics;
- применение события к локальному состоянию.

Именно это различие и задает правильную зону ответственности `SyncEngine`.

## `SyncError`: что означает эта ошибка

`SyncError` это единый внешний тип ошибок sync-контура.

Сейчас он содержит:

- `Store(StoreError)`
- `SerializePayload(serde_json::Error)`
- `DeserializePayload(serde_json::Error)`

### Как понимать эти ошибки

Важно различать:

- `SyncMessageOutcome` это нормализованный результат обработки сообщения;
- `SyncError` это уже сбой самого sync-пайплайна или его инфраструктурных зависимостей.

То есть:

- `Malformed`, `Invalid`, `Replay`, `Deferred` и `Unauthorized` это не `SyncError`;
- это штатные outcomes.

А вот `SyncError` означает:

- не удалось сериализовать или десериализовать payload на внутреннем техническом уровне;
- или store boundary сломался так, что sync не может продолжать работу как задумано.

Это очень важная граница для внешнего кода.

## Как верхние слои должны использовать этот модуль

Правильный сценарий использования обычно такой.

### Для обычного ingest

1. transport-слой получает `ReceivedEmail`;
2. верхний слой собирает batch;
3. создает `SyncEngine::new(&store)`;
4. вызывает `ingest_batch(batch)`;
5. анализирует `SyncReport`.

### Для повторной обработки

1. внешний слой знает, что есть deferred queue;
2. создает `SyncEngine::new(&store)`;
3. вызывает `reprocess_deferred()`;
4. получает `SyncReport`;
5. при необходимости агрегирует результат дальше.

Именно так сейчас его использует `app-core` для use case `reprocess_deferred_events`.

## Что этот модуль специально не делает

Чтобы не было ложных ожиданий, важно зафиксировать границы.

Этот модуль сейчас не делает следующего:

- не получает письма из mailbox сам;
- не открывает IMAP/SMTP соединения;
- не отдает UI-friendly diagnostics DTO;
- не является Tauri boundary;
- не управляет runtime lifecycle;
- не скрывает полностью transport/protocol distinction от верхнего слоя.

То есть это sync/apply layer, а не универсальный backend container.

## Что считать стабильным контрактом здесь

Если смотреть глазами потребителя crate, то наиболее значимыми частями интерфейса являются:

1. `SyncEngine::new(...)`;
2. поведение `ingest_batch(...)`;
3. поведение `reprocess_deferred()`;
4. `SyncReport` как внешний результат sync-операции;
5. набор и смысл вариантов `SyncMessageOutcome`;
6. различие между штатным outcome и настоящим `SyncError`.

Изменение этих элементов почти наверняка требует согласованных правок в:

- `liveletters-app-core`;
- `liveletters-diagnostics`;
- backend app;
- sync-тестах и диагностическом контуре.
