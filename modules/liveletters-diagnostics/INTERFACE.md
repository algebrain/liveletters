# `liveletters-diagnostics` INTERFACE

## Назначение

`liveletters-diagnostics` это внешний программный интерфейс диагностического слоя LiveLetters.

Если смотреть на уже пройденные модули по порядку:

- `liveletters-sync` умеет обработать входящие письма и записать технические следы в store;
- `liveletters-diagnostics` не выполняет sync сам, а читает эти следы и собирает из них осмысленный диагностический срез.

Именно этот модуль отвечает на вопросы вида:

- в каком общем состоянии находится контур синхронизации;
- сколько было applied, malformed, invalid, replay и unauthorized случаев;
- какие конкретно сообщения проблемные;
- какие события не применились;
- что сейчас лежит в deferred queue;
- что сейчас висит в outbox.

То есть это не логгер и не transport-движок, а слой чтения и агрегирования технического состояния.

## Где находится интерфейс

- crate: `liveletters-diagnostics`
- точка подключения: `src/lib.rs`

Наружу модуль экспортирует:

- `DiagnosticsReader`
- `DiagnosticsSnapshot`
- `SyncHealth`
- `HealthStatus`
- `RawMessageDiagnostic`
- `RawEventDiagnostic`
- `DeferredEventDiagnostic`
- `OutboxDiagnostic`
- `DiagnosticsError`

## Что считается внешним интерфейсом этого модуля

С практической точки зрения внешний интерфейс `liveletters-diagnostics` это:

1. `DiagnosticsReader::new(store)`;
2. метод `build_snapshot()`;
3. `DiagnosticsSnapshot` как полный диагностический срез;
4. `SyncHealth` как агрегированная health summary;
5. набор diagnostic DTO для сообщений, событий, deferred queue и outbox;
6. `DiagnosticsError` как единый внешний тип ошибки этого слоя.

Именно этим интерфейсом пользуются:

- backend app;
- диагностические integration tests;
- любой верхний Rust-код, который хочет получить технический срез без прямой работы с `Store`.

## Главный объект: `DiagnosticsReader`

### Зачем нужен `DiagnosticsReader`

`DiagnosticsReader` это фасад диагностического чтения.

Он нужен затем, чтобы верхний слой не делал все это сам:

- читать raw message journal;
- читать raw event journal;
- читать deferred queue;
- читать outbox;
- считать агрегаты;
- выводить health status;
- маскировать чувствительные данные в preview-полях.

Идея интерфейса такая:

- внешний слой дает `Store`;
- `DiagnosticsReader` сам собирает из него цельный диагностический снимок.

### Как создается `DiagnosticsReader`

`DiagnosticsReader::new(store: &Store) -> DiagnosticsReader`

Это означает:

- diagnostics работает поверх уже открытого store;
- жизненный цикл хранилища контролируется внешним слоем;
- модуль сам не открывает базу и не владеет ею.

Такой контракт согласован с остальными библиотеками нижнего и среднего уровня.

## Главная операция: `build_snapshot()`

### Что делает эта функция

`build_snapshot() -> Result<DiagnosticsSnapshot, DiagnosticsError>`

Это основная точка входа всего модуля.

Она:

1. читает журналы и очереди из store;
2. считает агрегированные counters;
3. определяет общий health status;
4. строит sanitизированные preview-значения;
5. возвращает единый `DiagnosticsSnapshot`.

То есть внешний код не должен самостоятельно собирать несколько кусков диагностики из разных store-методов. Он получает уже готовый срез.

### Почему это важно

Потому что diagnostics API должен быть:

- компактным;
- стабильным;
- пригодным для backend boundary;
- независимым от знания внутреннего расположения технических таблиц.

`build_snapshot()` и есть такой целевой интерфейс.

## `DiagnosticsSnapshot`: что это за тип

`DiagnosticsSnapshot` это полный диагностический снимок текущего состояния.

Он собирает вместе:

- `sync_health`
- `raw_messages`
- `raw_events`
- `deferred_events`
- `outbox_entries`

Смысл этого типа:

- не заставлять верхний слой отдельно запрашивать по кускам каждую категорию;
- дать один стабильный объект, который можно дальше маппить в backend DTO или использовать в тестах.

### Что с ним можно делать

Создается через:

- `DiagnosticsSnapshot::new(...)`

Читается через:

- `sync_health()`
- `raw_messages()`
- `raw_events()`
- `deferred_events()`
- `outbox_entries()`

То есть snapshot это read-only контейнер уже собранной диагностики.

## `SyncHealth`: зачем нужен этот тип

`SyncHealth` это агрегированная сводка состояния sync-контура.

Его смысл:

- дать короткий и пригодный для UI или backend status endpoint ответ на вопрос “насколько здоров sync-контур сейчас”.

Он хранит:

- `status`
- `applied_messages`
- `duplicate_messages`
- `replayed_messages`
- `unauthorized_messages`
- `invalid_messages`
- `malformed_messages`
- `deferred_events`
- `pending_outbox`

То есть `SyncHealth` это не детальный журнал, а summary-слой поверх него.

## `HealthStatus`: что означает этот enum

Сейчас наружу экспортируются два состояния:

- `Healthy`
- `Degraded`

Это бинарная, deliberately-conservative health model.

Смысл такой:

- `Healthy` означает, что diagnostics не видит проблемных признаков, которые сейчас считаются деградацией;
- `Degraded` означает, что такие признаки есть.

На текущем этапе health policy минимальна, но уже полезна для статуса backend и diagnostic UI.

## Как сейчас определяется здоровье контура

Из кода текущего модуля следует такая логика:

- `malformed`;
- `unauthorized`;
- `invalid`;
- наличие deferred queue

считаются достаточными сигналами для `Degraded`.

При этом:

- `duplicate`;
- `replay`

сами по себе не переводят статус в `Degraded`.

Это важная часть внешне наблюдаемого поведения diagnostics-модуля.

## `RawMessageDiagnostic`: зачем нужен этот тип

Это диагностическое представление одного входящего сообщения.

Он хранит:

- `message_id`
- `status`
- `preview`

Смысл этого типа:

- не возвращать наружу голый `RawMessageRecord` из store;
- а дать уже подготовленное диагностическое представление.

Особенно важно поле:

- `preview`

Оно специально существует, чтобы:

- показать оператору или UI краткий фрагмент содержимого;
- не вытаскивать наружу целиком все сырое письмо в обычный диагностический поток.

## `RawEventDiagnostic`: зачем нужен этот тип

Это диагностическое представление одного извлеченного события.

Он хранит:

- `event_id`
- `event_type`
- `resource_id`
- `apply_status`
- `failure_reason`

Смысл этого типа:

- показать уже не транспортный слой, а уровень применимости доменного события;
- дать ответ на вопрос “что именно пошло не так с конкретным событием”.

Это особенно важно там, где одного списка проблемных писем недостаточно.

## `DeferredEventDiagnostic`: зачем нужен этот тип

Это представление события, которое пока остается в deferred queue.

Он хранит:

- `event_id`
- `event_type`
- `reason`

Смысл этого типа:

- явно показать, что событие не потеряно;
- но и не применено;
- и по какой причине оно еще отложено.

Это важная часть diagnostics, потому что deferred events в LiveLetters считаются нормальным рабочим механизмом для out-of-order доставки, а не просто “ошибками”.

## `OutboxDiagnostic`: зачем нужен этот тип

Это представление одной записи outbox в диагностическом слое.

Он хранит:

- `event_id`
- `event_type`
- `resource_id`
- `preview`

Смысл этого типа:

- показать, что сейчас ожидает отправки;
- не возвращать наружу сырое тело сообщения как есть;
- дать краткий и безопасный preview.

То есть diagnostics в этом случае работает и для входящего, и для исходящего технического контура.

## Почему здесь используются отдельные diagnostic DTO

Важно понимать, что `liveletters-diagnostics` не просто прокидывает store records наружу.

Он делает дополнительную работу:

- агрегирует counters;
- определяет health status;
- sanitизирует preview;
- переводит низкоуровневые записи в более удобную диагностическую форму.

То есть diagnostic DTO существуют затем, чтобы backend app и diagnostic UI не были привязаны напрямую к форме store records.

## Санитизация preview: зачем она нужна

Одна из важных обязанностей модуля сейчас состоит в том, чтобы не тащить чувствительные данные в диагностический вывод без фильтрации.

Поэтому preview-поля:

- обрезаются по длине;
- проходят маскирование email-подобных токенов.

Смысл этого поведения:

- diagnostics должен быть полезным;
- но не должен без нужды копировать полные адреса и длинные сырые payload в обычный обзорный слой.

Это важная часть внешнего контракта модуля, потому что верхний слой может рассчитывать на уже sanitизированный preview.

## Где проходит граница между `sync`, `store` и `diagnostics`

Это важное различие.

### `store`

Отвечает за хранение:

- raw message journal;
- raw event journal;
- deferred queue;
- outbox.

### `sync`

Отвечает за то, как эти журналы появляются и обновляются:

- пишет `raw_messages`;
- пишет `raw_events`;
- решает, что `applied`, `invalid`, `replay`, `deferred`, `unauthorized` и так далее.

### `diagnostics`

Отвечает за чтение и интерпретацию уже накопленного технического следа:

- считает counters;
- строит health summary;
- формирует понятные диагностические представления.

То есть diagnostics не создает технические следы, а превращает их в наблюдаемую картину состояния.

## `DiagnosticsError`: что означает эта ошибка

`DiagnosticsError` это единый внешний тип ошибок диагностического слоя.

Сейчас он содержит:

- `Store(StoreError)`

### Как его понимать

Это означает, что модуль diagnostics сейчас концептуально read-only и зависит только от чтения store.

Если произошел `DiagnosticsError`, это не значит, что health “плохой”. Это значит, что сам диагностический срез не удалось корректно собрать из-за проблемы в нижнем persistence boundary.

То есть нужно различать:

- `HealthStatus::Degraded` как нормальное диагностируемое состояние системы;
- `DiagnosticsError` как сбой получения диагностики.

Это важное различие для backend boundary.

## Как верхние слои должны использовать этот модуль

Правильный сценарий использования обычно такой:

1. внешний слой получает `Store`;
2. создает `DiagnosticsReader::new(&store)`;
3. вызывает `build_snapshot()`;
4. получает `DiagnosticsSnapshot`;
5. дальше либо отдает его напрямую в Rust boundary, либо маппит в backend DTO.

Именно так сейчас его использует backend app, когда строит:

- `SyncStatusDto`;
- список incoming failures;
- список event failures.

То есть `liveletters-diagnostics` это промежуточный, но важный слой между техническим store/sync-контуром и внешним backend API.

## Что этот модуль специально не делает

Чтобы не было ложных ожиданий, важно зафиксировать границы.

Этот модуль сейчас не делает следующего:

- не запускает sync;
- не получает письма;
- не применяет события;
- не открывает store сам;
- не является Tauri boundary;
- не решает, как именно DTO должны выглядеть для frontend.

Иными словами:

- это диагностический read-model слой;
- но не runtime orchestration и не transport layer.

## Что считать стабильным контрактом здесь

Если смотреть глазами потребителя crate, то наиболее значимыми частями интерфейса являются:

1. `DiagnosticsReader::new(...)`;
2. `build_snapshot()`;
3. структура `DiagnosticsSnapshot`;
4. структура `SyncHealth` и логика `HealthStatus`;
5. формы diagnostic DTO для messages, events, deferred queue и outbox;
6. гарантия sanitизированных preview-полей;
7. различие между `DiagnosticsError` и обычным `Degraded` состоянием.

Изменение этих элементов почти наверняка требует согласованных правок в:

- backend app;
- frontend diagnostic boundary;
- tests для diagnostic snapshot и status API.
