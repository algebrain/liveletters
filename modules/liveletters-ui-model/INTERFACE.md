# `liveletters-ui-model` INTERFACE

## Назначение

`liveletters-ui-model` это внешний программный интерфейс слоя представления на frontend.

Если смотреть на уже пройденные frontend-модули:

- `liveletters-frontend-api` получает данные из backend и нормализует transport/integration boundary;
- `liveletters-ui-model` берет эти данные и превращает их в форму, удобную для экранов и app-store;
- уже поверх этого frontend app решает, как хранить состояние и как рендерить UI.

То есть этот модуль нужен не для вызова backend и не для отрисовки компонентов, а для следующей задачи:

- взять backend DTO или state slice;
- выразить его в screen-friendly форме;
- сделать это чистыми и предсказуемыми функциями.

Это и есть основная роль view-model слоя.

## Где находится интерфейс

- namespace: `liveletters.ui-model.core`

Именно этот namespace образует внешний интерфейс модуля.

## Что считается внешним интерфейсом этого модуля

С практической точки зрения внешний интерфейс `liveletters-ui-model` это:

1. `module-info`;
2. `feed-view-model`;
3. `post-thread-view-model`;
4. `sync-status-view-model`;
5. `incoming-failures-view-model`;
6. `event-failures-view-model`.

Именно через эти функции frontend app должен преобразовывать backend DTO в формы, пригодные для экранов.

## Главная идея модуля

Этот модуль существует затем, чтобы не смешивать три разные задачи в одном месте:

1. получение данных из backend;
2. хранение и orchestration состояния;
3. подготовку данных для отображения.

`liveletters-ui-model` отвечает только за третью задачу.

То есть:

- backend DTO приходит из `frontend-api`;
- app-store хранит данные и решает, когда что пересчитать;
- `ui-model` дает чистые функции преобразования в view model.

Это важно, потому что без такого слоя frontend быстро получает:

- экраны, в которых намешаны transport DTO и presentation logic;
- дублирование форматирования;
- слабую тестируемость экранной логики.

## Принцип работы: только чистые функции

Внешний интерфейс модуля сейчас состоит только из чистых функций.

Смысл этого решения:

- функция получает данные;
- возвращает новую структуру;
- не вызывает backend;
- не меняет глобальное состояние;
- не зависит от runtime;
- не рендерит UI.

Это и есть ключевой контракт `liveletters-ui-model`.

Если функция в этом модуле начнет:

- вызывать `invoke`;
- подписываться на события;
- дергать `atom`;
- рендерить hiccup;

она перестанет быть частью правильного интерфейса этого слоя.

## `module-info`

### Зачем нужна эта функция

`module-info` возвращает служебную метаинформацию:

- `{:module :liveletters-ui-model :language :cljc}`

Это не presentation use case, а идентификация модуля.

## `feed-view-model`: зачем нужна эта функция

### Что принимает

Функция принимает последовательность post DTO.

Это могут быть:

- данные, полученные из `get-home-feed!`;
- или уже сохраненный slice app-store.

### Что возвращает

Она возвращает map такой общей формы:

- `:items`
- `:empty?`

`items` содержит уже подготовленные элементы для экрана ленты.

### Что делает внутри

Сейчас функция:

- сортирует посты по `:post_id` по убыванию;
- строит для каждого поста компактную view model;
- вычисляет `:empty?`.

### Как понимать возвращаемые поля

Каждый элемент `:items` содержит:

- `:id`
- `:title`
- `:meta`
- `:hidden?`

Смысл этих полей:

- `:id` нужен как стабильный идентификатор элемента списка;
- `:title` это отображаемое содержимое поста в компактной форме;
- `:meta` это уже собранная строка для вторичной информации;
- `:hidden?` это presentation-friendly флаг, который UI может напрямую использовать.

### Зачем это вообще нужно

Чтобы экран ленты не занимался сам:

- сортировкой;
- склейкой meta-строки;
- извлечением полей из сырого backend DTO.

`feed-view-model` делает из transport/read DTO уже экранную форму.

## `post-thread-view-model`: зачем нужна эта функция

### Что принимает

Функция принимает thread DTO с двумя ключевыми частями:

- `:post`
- `:comments`

Это обычно результат backend query `get_post_thread`.

### Что возвращает

Она возвращает структуру с ключами:

- `:post`
- `:comments`
- `:comment-count`

То есть уже готовую основу для экрана поста и комментариев.

### Что делает внутри

Сейчас она:

- преобразует пост в компактную presentation-форму;
- преобразует комментарии в список screen-friendly элементов;
- считает количество комментариев.

### Как понимать возвращаемые данные

`post` содержит:

- `:id`
- `:title`
- `:hidden?`

Каждый комментарий содержит:

- `:id`
- `:body`
- `:author`
- `:reply-to`

Смысл этого преобразования:

- UI не должен знать исходные backend-имена вроде `:comment_id` или `:parent_comment_id`;
- он получает уже presentation-level форму, которая ближе к тому, как экран реально мыслит данными.

### Почему это важно

Без такого слоя thread screen будет сам:

- переименовывать поля;
- считать комментарии;
- тащить в себя mapping rules.

`post-thread-view-model` выносит это в отдельный и тестируемый boundary.

## `sync-status-view-model`: зачем нужна эта функция

### Что принимает

Функция принимает sync status DTO после normalization из `liveletters-frontend-api`.

То есть на вход ей дается уже transport-normalized статус backend.

### Что возвращает

Она возвращает структуру:

- `:label`
- `:tone`
- `:details`

Это уже presentation-layer форма для status panel.

### Что означает результат

`label` это человекочитаемое состояние:

- например `Healthy`
- или `Degraded`

`tone` это UI-friendly семантика:

- `:positive`
- `:warning`
- `:neutral`

`details` содержит уже presentation-level counters:

- `:applied`
- `:duplicates`
- `:replays`
- `:unauthorized`
- `:invalid`
- `:malformed`
- `:deferred`
- `:outbox`

### Зачем это нужно

Чтобы экран диагностики или status widget не занимался сам:

- сопоставлением backend status и label;
- вычислением tone;
- переименованием всех счетчиков.

То есть это чистая функция presentation mapping для diagnostic summary.

## `incoming-failures-view-model`: зачем нужна эта функция

### Что принимает

Функция принимает список incoming failure DTO.

Это обычно результат `list-incoming-failures!`.

### Что возвращает

Она возвращает vector элементов с ключами:

- `:id`
- `:kind`
- `:preview`

### Что она делает

Сейчас она:

- переводит backend status в keyword через `kind`;
- сохраняет id сообщения;
- сохраняет preview для отображения.

### Зачем это нужно

Потому что экрану диагностики удобнее работать не с raw backend-формой, а с уже немного осмысленным presentation representation.

Смысл `kind` здесь особенно важен:

- это более frontend-friendly форма категории сбоя;
- UI может дальше маппить ее в стиль, иконку, текст или фильтрацию.

## `event-failures-view-model`: зачем нужна эта функция

### Что принимает

Функция принимает список event failure DTO.

Это обычно нормализованный результат `list-event-failures!`.

### Что возвращает

Она возвращает vector элементов с ключами:

- `:id`
- `:event-type`
- `:resource-id`
- `:kind`
- `:reason`

### Что это означает

Это уже presentation-ready форма проблемных событий.

Смысл полей:

- `:id` это event identifier;
- `:event-type` это краткая классификация события;
- `:resource-id` помогает привязать проблему к ресурсу;
- `:kind` выражает apply-status в frontend-friendly форме;
- `:reason` это текстовая причина для диагностического отображения.

### Зачем это нужно

Экран диагностики событий не должен сам переименовывать backend DTO. Он должен получать уже понятную структуру, на которой можно строить список, фильтрацию и визуальное оформление.

## Почему эти функции не рендерят UI сами

Это важная граница.

`liveletters-ui-model` не должен превращаться в библиотеку компонентов.

Если функция уже знает:

- какие HTML-теги рисовать;
- какие CSS-классы использовать;
- как именно расположить блоки;

это уже не view model, а presentation component.

Правильная роль этого модуля уже:

- выбрать данные;
- переименовать поля;
- собрать presentation-level структуру;
- но не рендерить.

То есть UI остается задачей:

- `liveletters-ui-kit`;
- frontend app.

## Где проходит граница между `frontend-api`, `ui-model`, `ui-kit` и frontend app

Это одна из ключевых архитектурных границ frontend-части.

### `frontend-api`

Отвечает за:

- вызов backend;
- подписку на backend events;
- normalization transport DTO и ошибок.

### `ui-model`

Отвечает за:

- mapping DTO -> view model;
- presentation-level сортировку;
- presentation-level grouping;
- подготовку screen-friendly данных.

### `ui-kit`

Отвечает за:

- reusable UI primitives;
- компоненты и визуальные блоки.

### frontend app

Отвечает за:

- app-store;
- orchestration сценариев;
- когда и какие view model считать;
- как их скармливать UI-компонентам.

То есть `ui-model` находится строго между “данные пришли” и “компоненты начали рисоваться”.

## Как верхние слои должны использовать этот модуль

Правильный сценарий использования обычно такой:

1. frontend получает нормализованный backend DTO из `frontend-api`;
2. сохраняет его в app-store или передает дальше;
3. вызывает функцию из `liveletters.ui-model.core`;
4. получает screen-friendly структуру;
5. уже эту структуру отдает в компонентный слой.

То есть компоненты по возможности должны зависеть именно от view model, а не от сырых backend DTO.

## Что этот модуль специально не делает

Чтобы не было ложных ожиданий, важно зафиксировать границы.

Этот модуль сейчас не делает следующего:

- не вызывает backend;
- не подписывается на события;
- не хранит app-state;
- не рендерит hiccup;
- не содержит reusable UI primitives;
- не маршрутизирует пользователя.

Иными словами:

- это слой presentation mapping;
- но не integration layer, не app-store и не component library.

## Что считать стабильным контрактом здесь

Если смотреть глазами потребителя модуля, то наиболее значимыми частями интерфейса являются:

1. набор экспортируемых view-model функций;
2. формы возвращаемых структур;
3. sorting/mapping rules внутри этих функций;
4. договоренность, что функции остаются чистыми;
5. граница, по которой backend DTO превращаются в screen-friendly данные.

Текущее дополнение к этому контракту:

6. `settings-form-view-model` как экранная форма данных для initial setup и settings page.

Изменение этих элементов почти наверняка требует согласованных правок в:

- frontend app;
- `liveletters-ui-kit`, если компоненты ожидают конкретную форму props;
- tests для presentation mapping.
