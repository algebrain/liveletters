# `liveletters-store` INTERFACE

## Назначение

`liveletters-store` это внешний программный интерфейс локального хранилища LiveLetters.

Если смотреть на уже пройденные модули по порядку:

- `liveletters-domain` описывает форму предметной модели;
- `liveletters-protocol` описывает форму технического сообщения;
- `liveletters-store` отвечает за то, как текущее локальное состояние и технические журналы живут на диске или в памяти.

Это не просто “обертка над SQLite”, а boundary, через который верхние слои читают и пишут:

- материализованные посты и комментарии;
- outbox;
- журнал сырых входящих сообщений;
- журнал сырых событий;
- очередь отложенных событий.

Модуль не знает про frontend, Tauri, IMAP/SMTP и экранную модель. Его задача уже и приземленнее:

- открыть хранилище;
- обеспечить базовую схему;
- дать понятные операции записи и чтения.

## Где находится интерфейс

- crate: `liveletters-store`
- точка подключения: `src/lib.rs`

Наружу модуль экспортирует:

- основной объект `Store`;
- конфигурацию путей `StorePaths`;
- тип ошибки `StoreError`;
- record-типы, через которые верхние слои обмениваются данными с persistence-контуром.

## Что считается внешним интерфейсом этого модуля

С практической точки зрения внешний интерфейс `liveletters-store` это:

1. способы открыть store;
2. способ определить стандартные пути хранения;
3. набор record-типов для materialized state и технических журналов;
4. методы чтения и записи в `Store`;
5. `StoreError` как единый внешний тип ошибок persistence-слоя.

Именно этим API пользуются:

- `liveletters-app-core`;
- `liveletters-sync`;
- `liveletters-diagnostics`;
- backend app.

## Главный объект: `Store`

### Зачем нужен `Store`

`Store` это фасад над локальным SQLite-хранилищем.

Он нужен затем, чтобы верхние слои не работали напрямую с:

- SQL-строками;
- открытием соединения;
- созданием каталогов;
- начальной инициализацией схемы.

Идея этого интерфейса простая:

- внешний код работает не с базой как таковой;
- а с набором осмысленных операций уровня “сохранить пост”, “получить комментарии поста”, “посмотреть outbox”, “посмотреть deferred events”.

### Как открывается `Store`

У `Store` есть три основных способа открытия.

#### `Store::open_default()`

Это основной путь для обычного runtime-сценария.

Он:

- определяет домашний каталог пользователя;
- строит стандартный путь хранения LiveLetters;
- открывает SQLite по этому пути;
- гарантирует инициализацию текущей схемы.

То есть это “подними обычное локальное хранилище приложения”.

#### `Store::open_for_home_dir(home_dir)`

Это явный файловый режим открытия от заданного home-каталога.

Он нужен, когда вызывающий код хочет запустить LiveLetters так, как будто:

- `HOME == home_dir`

Практически это означает:

- data dir будет вычислен как `home_dir/.liveletters`;
- SQLite-файл будет открыт по стандартному пути внутри этого каталога;
- если самого `home_dir` еще нет, нужное дерево каталогов будет создано автоматически.

#### `Store::open_in_memory()`

Это временное хранилище в памяти.

Оно полезно:

- для тестов;
- для smoke-сценариев;
- для локального прогона без записи на диск.

Смысл этой точки входа в том, что можно поднимать весь persistence contour без файловой системы.

#### `Store::open_at(path)`

Это явный режим открытия по заданному пути.

Он нужен:

- для интеграционных тестов;
- для tooling и служебных сценариев;
- везде, где вызывающий код сам хочет контролировать физическое расположение базы.

Если каталога по пути еще нет, store создает его сам.

### Что важно про инициализацию

Все три способа открытия гарантируют одно и то же:

- схема текущей базы будет инициализирована.

То есть вызывающий код не должен отдельно думать о bootstrap SQL. Для него это уже часть контракта store.

## `StorePaths`: зачем нужен этот тип

`StorePaths` это отдельный тип для вычисления стандартных путей хранения LiveLetters.

Он нужен затем, чтобы логика “где лежат данные приложения” не была размазана по верхним слоям.

### Что он описывает

Сейчас `StorePaths` хранит:

- `data_dir`
- `database_path`
- `runtime_log_dir`

То есть:

- каталог приложения внутри домашней директории;
- путь к SQLite-файлу внутри этого каталога.
- каталог временных runtime logs внутри того же data-home.

### Как его использовать

Есть два основных способа.

#### `StorePaths::for_home_dir(home_dir)`

Это явный способ построить пути, если вызывающий код уже знает домашний каталог.

Полезно:

- в тестах;
- в сценариях с временным `HOME`;
- когда путь контролируется снаружи.

#### `StorePaths::from_environment()`

Это способ взять домашний каталог из окружения:

- `HOME`
- или `USERPROFILE`

И уже на его основе построить стандартные пути LiveLetters.

После этого можно читать:

- `data_dir()`
- `database_path()`
- `runtime_log_dir()`

### Почему это важно для внешнего интерфейса

Потому что `StorePaths` задает стандартную и повторяемую политику расположения данных. Верхний слой не должен вручную собирать путь вида `~/.liveletters/liveletters.sqlite3` или `~/.liveletters/runtime-logs` в нескольких местах независимо.

## Какие данные store хранит

На уровне внешнего API `liveletters-store` работает не с абстрактными таблицами, а с несколькими категориями данных.

### 1. Материализованное состояние

Сюда сейчас относятся:

- посты;
- комментарии.

Это то состояние, которое потом читает `app-core` и отдает в UI как feed/thread read models.

### 2. Исходящий контур

Сюда относится:

- outbox.

То есть сообщения, которые backend уже сформировал и поставил в очередь исходящей доставки.

### 3. Входящий технический журнал

Сюда относятся:

- `raw_messages`;
- `raw_events`.

Это уже не materialized state, а следы работы sync-контура:

- какое письмо пришло;
- какое событие из него было извлечено;
- был ли apply успешным;
- если нет, то почему.

### 4. Очередь отложенных событий

Сюда относятся:

- `deferred_events`.

Это отдельный контур для событий, которые еще нельзя применить прямо сейчас, но которые не нужно терять.

## Record-типы: зачем они вообще нужны

`liveletters-store` экспортирует набор record-типов:

- `PostRecord`
- `CommentRecord`
- `OutboxRecord`
- `RawMessageRecord`
- `RawEventRecord`
- `DeferredEventRecord`

Смысл этих типов в том, что store boundary работает не с доменными сущностями напрямую, а с persistence-friendly структурами.

Это важное различие:

- `Post` из `liveletters-domain` это доменная сущность;
- `PostRecord` из `liveletters-store` это форма записи в локальном хранилище.

То есть record-типы это не дублирование без смысла, а явное разделение:

- доменная модель;
- persistence model.

## Что означает каждый record-тип

### `PostRecord`

Это сохраненный материализованный пост.

Он нужен, когда верхний слой хочет:

- записать текущее состояние поста в базу;
- прочитать пост из materialized state.

Он хранит:

- идентификаторы;
- время;
- body;
- visibility как строку;
- hidden-флаг.

### `CommentRecord`

Это сохраненный материализованный комментарий.

Он нужен для тех же целей, что и `PostRecord`, но для комментариев, включая поддержку `parent_comment_id`.

### `OutboxRecord`

Это запись о сообщении, которое уже подготовлено к отправке.

Он хранит:

- `event_id`
- `event_type`
- `resource_id`
- `message_body`

Главный смысл:

- приложение уже сформировало техническое сообщение;
- его нужно держать в очереди исходящей доставки.

### `RawMessageRecord`

Это запись о сыром входящем сообщении.

Она нужна для:

- диагностики;
- повторного анализа;
- учета статуса обработки транспортного письма.

Хранит:

- `message_id`
- `raw_message`
- `status`

### `RawEventRecord`

Это запись о событии, извлеченном из входящего сообщения.

Она важна для sync и diagnostics, потому что хранит:

- `event_id`
- `event_type`
- `resource_id`
- `payload_json`
- `apply_status`
- `failure_reason`

Именно здесь отражается, например:

- что событие было `applied`;
- или оказалось `invalid`;
- или было `replay`;
- или оказалось `unauthorized`.

### `DeferredEventRecord`

Это запись об отложенном событии.

Она нужна, когда событие:

- не является malformed;
- не является окончательно отклоненным;
- но пока не может быть применено из-за нехватки зависимостей.

Такой record хранит:

- `event_id`
- `event_type`
- `reason`
- `payload_json`

То есть это “не забудь попробовать применить это позже”.

## Какие операции предоставляет `Store`

Ниже важно не просто перечислить методы, а понимать, для чего предназначена каждая группа операций.

### Операции с постами

#### `save_post_record(&PostRecord)`

Сохраняет или перезаписывает материализованный пост.

Когда используется:

- после создания поста;
- после скрытия поста;
- после любого другого доменного перехода, который меняет текущее состояние поста.

#### `list_posts()`

Возвращает все сохраненные посты в materialized state.

Когда используется:

- для построения ленты;
- для read-only запросов верхнего уровня.

#### `get_post_record(post_id)`

Возвращает один пост по идентификатору или `None`, если его нет.

Когда используется:

- перед созданием зависимых сущностей;
- при чтении конкретного поста;
- в командах, которым нужно сначала проверить существование поста.

### Операции с комментариями

#### `save_comment_record(&CommentRecord)`

Сохраняет или перезаписывает материализованный комментарий.

Когда используется:

- после создания комментария;
- после редактирования;
- после скрытия;
- после применения входящего события.

#### `list_comments_for_post(post_id)`

Возвращает все комментарии конкретного поста.

Это основной низкоуровневый источник для сборки thread view.

#### `get_comment_record(comment_id)`

Возвращает один комментарий по идентификатору.

Нужен, когда верхний слой хочет:

- проверить существование комментария;
- получить текущее состояние перед доменным переходом;
- валидировать действия редактирования или скрытия.

### Операции с outbox

#### `save_outbox_record(&OutboxRecord)`

Сохраняет сообщение в исходящую очередь.

Когда используется:

- после того как `app-core` собрал protocol message;
- когда нужно зафиксировать, что сообщение готово к дальнейшей доставке.

#### `list_outbox_records()`

Возвращает текущую очередь исходящих сообщений.

Нужен:

- для отправки;
- для диагностики;
- для status summary, где показывается pending outbox.

### Операции с сырыми входящими сообщениями

#### `save_raw_message_record(&RawMessageRecord)`

Сохраняет транспортное сообщение и итог его обработки.

Используется sync-слоем при ingest.

#### `list_raw_message_records()`

Возвращает журнал сырых входящих сообщений.

Нужен:

- для диагностики;
- для подсчета sync health;
- для анализа проблемных входящих писем.

### Операции с сырыми событиями

#### `save_raw_event_record(&RawEventRecord)`

Сохраняет событие, извлеченное из payload, вместе со статусом применения.

Это одна из самых важных операций sync-контура, потому что именно она фиксирует:

- что событие увидели;
- что попытались с ним сделать;
- к какому результату это привело.

#### `list_raw_event_records()`

Возвращает журнал событийного ingest-контура.

Нужен в первую очередь для diagnostics и backend status API.

#### `has_raw_event(event_id)`

Проверяет, есть ли уже событие с таким `event_id`.

Смысл этой операции:

- поддержать duplicate detection;
- не заставлять верхний слой самому писать SQL для проверки идемпотентности.

### Операции с deferred events

#### `save_deferred_event_record(&DeferredEventRecord)`

Сохраняет событие в очередь отложенной обработки.

Когда используется:

- если событие само по себе валидно;
- но пока не может быть применено, например из-за отсутствия зависимого поста или комментария.

#### `list_deferred_event_records()`

Возвращает всю текущую очередь deferred events.

Нужен для повторной переобработки.

#### `delete_deferred_event_record(event_id)`

Удаляет событие из deferred queue.

Когда используется:

- после успешного применения;
- после окончательного вывода, что событие больше не должно оставаться отложенным.

## `StoreError`: что означает эта ошибка

`StoreError` это внешний тип ошибок persistence-слоя.

Сейчас он содержит:

- `Sqlite(rusqlite::Error)`
- `Io(std::io::Error)`
- `MissingHomeDirectory`

### Как понимать эти ошибки

#### `Sqlite(...)`

Означает проблему на уровне самой базы:

- ошибка открытия;
- ошибка SQL-запроса;
- ошибка чтения/записи.

#### `Io(...)`

Означает проблему файловой системы:

- не удалось создать каталог;
- не удалось получить доступ к пути;
- другая I/O ошибка вокруг базы.

#### `MissingHomeDirectory`

Означает, что стандартный путь хранения нельзя вычислить из окружения.

Это важный внешний контракт `open_default()` и `StorePaths::from_environment()`: они зависят от корректного `HOME` или `USERPROFILE`.

## Как верхние слои должны использовать этот модуль

Правильная схема использования обычно такая.

### Для application layer

`app-core` использует store как persistence boundary:

- берет текущие `PostRecord` и `CommentRecord`;
- обновляет их после доменных переходов;
- пишет сообщения в outbox.

### Для sync layer

`sync` использует store как ingest journal и materialized state:

- сохраняет `raw_messages`;
- сохраняет `raw_events`;
- проверяет duplicate через `has_raw_event`;
- пишет или удаляет `deferred_events`;
- обновляет materialized post/comment state.

### Для diagnostics layer

`diagnostics` использует store как источник технических срезов:

- читает raw message journal;
- читает raw event journal;
- читает deferred queue;
- читает outbox.

То есть store это общий persistence-фундамент для нескольких верхних модулей.

## Что этот модуль специально не делает

Чтобы не было ложных ожиданий, важно зафиксировать и ограничения.

Этот модуль сейчас не делает следующего:

- не принимает доменных решений;
- не знает, кто имеет право редактировать пост или комментарий;
- не строит protocol payload;
- не отправляет письма;
- не парсит входящие сообщения;
- не агрегирует sync health;
- не отдает UI-ready DTO;
- не предоставляет полноценный высокоуровневый transaction scripting API.

То есть это именно persistence boundary, а не application service и не sync engine.

## Что считать стабильным контрактом здесь

Если смотреть глазами потребителя crate, то наиболее значимыми частями интерфейса являются:

1. способы открытия `Store`;
2. `StorePaths` как политика стандартного размещения данных;
3. формы record-типов;
4. набор операций `Store` для materialized state, outbox, raw journal и deferred queue;
5. `has_raw_event(...)` как часть идемпотентного sync-контура;
6. `StoreError` как единый внешний тип ошибок persistence-слоя.

Текущее дополнение к этому контракту:

7. `UserSettingsRecord` и `MailSettingsRecord`;
8. операции чтения и записи локального settings contour.

Изменение этих элементов почти наверняка требует согласованных правок в:

- `liveletters-app-core`;
- `liveletters-sync`;
- `liveletters-diagnostics`;
- backend app;
- интеграционных тестах persistence и sync-пайплайна.
