# `liveletters-ui-kit` INTERFACE

## Назначение

`liveletters-ui-kit` это внешний программный интерфейс базовых UI-компонентов LiveLetters.

Если смотреть на уже пройденные frontend-модули:

- `liveletters-frontend-api` отвечает за интеграцию с backend;
- `liveletters-ui-model` отвечает за преобразование данных в screen-friendly view model;
- `liveletters-ui-kit` отвечает уже за следующий слой: из каких переиспользуемых визуальных primitives строится интерфейс.

Этот модуль не должен содержать:

- вызовы backend;
- глобальное состояние;
- маршрутизацию;
- orchestration экранов.

Его роль уже:

- дать повторно используемые компоненты;
- зафиксировать базовые props и поведение;
- обеспечить единообразие простых UI-элементов.

То есть это компонентный, а не интеграционный и не stateful слой.

## Где находится интерфейс

- namespace: `liveletters.ui-kit.core`

Именно этот namespace образует внешний интерфейс модуля.

## Что считается внешним интерфейсом этого модуля

С практической точки зрения внешний интерфейс `liveletters-ui-kit` это:

1. `module-info`;
2. `button`;
3. `text-input`;
4. `section`;
5. `loading-state`;
6. `empty-state`;
7. `error-state`.

Это небольшой набор primitives текущей реализации, из которых фронтенд может строить более крупные экраны и виджеты.

## Главная идея модуля

`liveletters-ui-kit` нужен затем, чтобы простые UI-элементы не создавались заново в каждом экране.

Без такого слоя фронтенд быстро получает:

- повторяющиеся кнопки и поля с разными мелкими отличиями;
- разные соглашения по class names;
- разнобой в props;
- размазанную accessibility-логику.

`ui-kit` делает обратное:

- собирает повторно используемые visual primitives в одном месте;
- задает им понятный внешний контракт;
- позволяет экранам зависеть от компонентных функций, а не от случайной локальной разметки.

## Принцип интерфейса

В текущей реализации внешний интерфейс модуля состоит из функций, возвращающих hiccup-структуры.

Это означает:

- внешний код передает props map;
- функция возвращает готовый описательный UI-узел;
- модуль не хранит состояние сам по себе;
- компоненты остаются простыми и композиционными.

То есть `ui-kit` в текущем виде это библиотека функциональных Reagent-совместимых компонентов.

## `module-info`

### Зачем нужна эта функция

`module-info` возвращает служебную метаинформацию:

- `{:module :liveletters-ui-kit :language :cljc}`

Это не UI-сценарий, а просто способ идентифицировать модуль.

## `button`: базовая кнопка

### Зачем нужен этот компонент

`button` это базовый primitive для кликабельных действий.

Смысл его вынесения в `ui-kit`:

- не создавать кнопки заново в каждом экране;
- держать общий формат props;
- централизованно задавать CSS class naming и базовое поведение.

### Что принимает

Компонент принимает props map с ключами:

- `:label`
- `:variant`
- `:on-click`
- `:disabled?`

### Что означают эти поля

`label`:

- текст кнопки.

`variant`:

- семантический вариант оформления;
- по умолчанию `:primary`.

`on-click`:

- callback, который будет вызван при клике, если он передан.

`disabled?`:

- флаг, отключающий кнопку;
- по умолчанию `false`.

### Что возвращает

Компонент возвращает hiccup для HTML `<button>`.

При этом он:

- всегда ставит `type="button"`;
- формирует CSS-класс вида `ll-button ll-button--<variant>`;
- выставляет `:disabled`, если нужно;
- добавляет `:on-click`, если callback был передан.

### Зачем это важно

С точки зрения внешнего интерфейса это означает:

- кнопка имеет предсказуемый contract;
- вариант оформления задается не произвольной версткой, а props;
- экранный код не должен повторять одинаковый шаблон разметки.

## `text-input`: базовое текстовое поле

### Зачем нужен этот компонент

`text-input` это primitive для текстового ввода.

Его задача:

- дать единый интерфейс простого текстового поля;
- собрать label и input в один reusable компонент;
- задать понятный contract controlled/read-only поведения.

### Что принимает

Компонент принимает props map с ключами:

- `:label`
- `:value`
- `:placeholder`
- `:on-change`

### Что означают эти поля

`label`:

- подпись поля.

`value`:

- текущее значение;
- по умолчанию `""`.

`placeholder`:

- placeholder текста;
- по умолчанию `""`.

`on-change`:

- callback изменения значения.

### Что важно в поведении

Если `on-change` не передан, компонент:

- делает поле read-only.

Это важная часть контракта, потому что она задает безопасное поведение для случаев, когда поле должно быть отображаемым, но не редактируемым.

### Что возвращает

Компонент возвращает hiccup-структуру, которая включает:

- label wrapper;
- подпись;
- `<input type="text">`.

При этом используются стабильные CSS-классы:

- `ll-field`
- `ll-field__label`
- `ll-input`

То есть компонент не только упрощает разметку, но и задает единый structural contract для styling.

## `section`: базовый секционный контейнер

### Зачем нужен этот компонент

`section` это layout primitive для простого блока с заголовком и содержимым.

Он нужен затем, чтобы экраны не повторяли один и тот же паттерн:

- секция;
- заголовок;
- body container.

### Что принимает

Компонент принимает props map с ключами:

- `:title`
- `:children`

### Что важно

Если `children` не передан, по умолчанию используется:

- `[]`

То есть компонент безопасен для пустого состояния.

### Что возвращает

Он возвращает hiccup-структуру:

- `<section>`
- заголовок `<h2>`
- контейнер для дочернего содержимого

с CSS-классами:

- `ll-section`
- `ll-section__title`
- `ll-section__body`

Смысл для внешнего кода:

- можно собирать экран из повторяемых секций;
- не размазывая layout boilerplate по компонентам приложения.

## `loading-state`: базовое состояние загрузки

### Зачем нужен этот компонент

`loading-state` нужен для единообразного отображения состояния “идет загрузка”.

Без него каждый экран может по-разному оформлять один и тот же паттерн.

### Что принимает

Props map с ключом:

- `:message`

### Что возвращает

Компонент возвращает блок с:

- CSS-классом `ll-state ll-state--loading`;
- `role="status"`;
- `aria-live="polite"`.

### Почему это важно

Это уже не просто “див с текстом”, а маленький accessibility-aware primitive.

С точки зрения интерфейса это означает:

- внешний код получает готовый паттерн состояния загрузки;
- не обязан заново помнить про базовые a11y-атрибуты.

## `empty-state`: базовое пустое состояние

### Зачем нужен этот компонент

`empty-state` нужен для отображения ситуации, когда данных нет, но это не ошибка и не загрузка.

### Что принимает

Props map с ключом:

- `:message`

### Что возвращает

Блок с CSS-классом:

- `ll-state ll-state--empty`

Смысл:

- единый primitive пустого состояния;
- без повторения однотипной разметки в экранах.

## `error-state`: базовое состояние ошибки

### Зачем нужен этот компонент

`error-state` нужен для единообразного отображения экранной ошибки.

### Что принимает

Props map с ключом:

- `:message`

### Что возвращает

Блок с:

- CSS-классом `ll-state ll-state--error`;
- `role="alert"`.

### Почему это важно

Смысл этого интерфейса:

- состояние ошибки оформляется единообразно;
- базовый accessibility contract задается на уровне ui-kit, а не дублируется в каждом месте.

## Почему этот модуль не должен заниматься данными

Это ключевая граница.

`liveletters-ui-kit` не должен знать:

- откуда пришли данные;
- что означает конкретный backend DTO;
- когда нужно запрашивать обновление;
- как устроен app-store.

Если компонент начинает понимать transport DTO или backend команды, это уже не reusable primitive, а часть более высокого слоя.

Правильная роль `ui-kit` уже:

- получать подготовленные props;
- возвращать UI primitive.

## Где проходит граница между `ui-kit`, `ui-model` и frontend app

Это одна из ключевых frontend-границ.

### `ui-kit`

Отвечает за:

- reusable компоненты;
- простые layout и state primitives;
- базовые a11y-aware визуальные блоки.

### `ui-model`

Отвечает за:

- mapping DTO -> view model;
- presentation-level данные;
- подготовку screen-friendly структур.

### frontend app

Отвечает за:

- app-store;
- orchestration;
- когда использовать тот или иной компонент;
- как связать view model и UI.

То есть:

- `ui-model` подготавливает данные;
- `ui-kit` подготавливает primitives;
- frontend app склеивает их в реальные экраны.

## Как верхние слои должны использовать этот модуль

Правильный сценарий использования обычно такой:

1. frontend app получает view model из `liveletters-ui-model`;
2. выбирает нужные компоненты из `liveletters.ui-kit.core`;
3. передает в них уже готовые props;
4. композитно собирает из них экран.

То есть экранный код должен по возможности зависеть:

- от view model для данных;
- от ui-kit для компонентов;
- а не рисовать все вручную каждый раз.

## Что этот модуль специально не делает

Чтобы не было ложных ожиданий, важно зафиксировать границы.

Этот модуль сейчас не делает следующего:

- не вызывает backend;
- не нормализует DTO;
- не хранит app-state;
- не содержит route-level или feature-level orchestration;
- не заменяет полноценный design system;
- не отвечает за глобальную тему или CSS pipeline сам по себе.

Иными словами:

- это компонентный primitive layer;
- но не integration layer, не state layer и не orchestration layer.

## Что считать стабильным контрактом здесь

Если смотреть глазами потребителя модуля, то наиболее значимыми частями интерфейса являются:

1. набор экспортируемых primitive-компонентов;
2. формы props, которые они принимают;
3. их базовое поведение;
4. их class naming contract;
5. базовые accessibility-атрибуты у state-компонентов.

Текущее дополнение к этому контракту:

6. `password-input` как базовое поле для SMTP и IMAP паролей.

Изменение этих элементов почти наверняка требует согласованных правок в:

- frontend app;
- tests на rendering и basic UI behavior;
- локальный CSS/стили, если они опираются на текущие `ll-*` class names.
