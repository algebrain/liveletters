# `liveletters-domain` INTERFACE

## Назначение

`liveletters-domain` это базовый предметный интерфейс LiveLetters.

Если смотреть на весь репозиторий сверху, именно этот модуль отвечает на вопрос:

- какие сущности вообще существуют в системе;
- какие у них устойчивые идентификаторы;
- какие минимальные инварианты считаются обязательными;
- какие доменные события выражают изменения состояния.

Это самый нижний и наименее зависимый программный интерфейс в проекте. Он не знает ничего про:

- Tauri;
- frontend;
- SQLite;
- IMAP/SMTP;
- MIME;
- diagnostics;
- runtime wiring.

То есть этот модуль не про хранение, транспорт или UI, а про саму форму предметной модели.

## Где находится интерфейс

- crate: `liveletters-domain`
- точка подключения: `src/lib.rs`

Наружу модуль экспортирует типы, из которых строится предметная модель:

- идентификаторы;
- value objects;
- сущности;
- доменные события;
- ошибки доменной валидации.

## Что считается внешним интерфейсом этого модуля

С практической точки зрения внешний интерфейс `liveletters-domain` это:

1. типы идентификаторов;
2. типы полезных значений вроде `PostBody`, `CommentBody`, `Timestamp`;
3. сущности `Post` и `Comment`;
4. доменные события `PostCreated`, `CommentCreated`, `PostHidden`, `CommentEdited`, `CommentHidden`;
5. перечисление `Visibility`;
6. ошибка `DomainError`.

Именно этими типами должны пользоваться верхние слои, когда хотят:

- создать доменную сущность;
- выразить доменное изменение;
- проверить, что входные данные вообще допустимы на уровне модели.

## Идентификаторы: зачем они нужны

В модуле есть отдельные типы идентификаторов:

- `PostId`
- `CommentId`
- `ResourceId`
- `AccountId`
- `EventId`

Их смысл не только в том, чтобы хранить строку, а в том, чтобы не смешивать разные идентичности между собой.

Например:

- `post_id` и `comment_id` обе могут быть строками;
- но на уровне доменной модели это разные сущности;
- значит, они должны быть разными типами.

Такой интерфейс полезен для верхних слоев, потому что он уменьшает риск случайно передать не тот идентификатор не туда.

### Что с ними можно делать

Каждый идентификатор создается через:

- `new(&str) -> Result<..., DomainError>`

Это означает, что внешний код не должен создавать такие значения как “любую строку”. Сначала строка проходит доменную проверку.

После создания идентификатор можно:

- хранить внутри сущностей;
- передавать в доменные события;
- читать обратно через `as_str()`.

### Какие ограничения накладываются

Сейчас инвариант у идентификаторов минимальный, но важный:

- идентификатор не может быть пустым;
- идентификатор не может состоять только из пробелов.

Если это правило нарушено, модуль возвращает:

- `DomainError::BlankIdentifier(...)`

То есть внешний интерфейс прямо говорит: пустые идентификаторы не являются допустимым состоянием модели.

## Value objects: зачем нужны `PostBody`, `CommentBody` и `Timestamp`

### `PostBody`

`PostBody` это не просто строка текста, а тип “тело поста”.

Его смысл:

- отделить текст поста от любых других строк;
- не позволить случайно построить пост с пустым содержимым;
- зафиксировать, что проверка допустимости текста относится к доменному уровню.

Создается через:

- `PostBody::new(&str) -> Result<PostBody, DomainError>`

После создания значение можно читать через:

- `as_str()`

### `CommentBody`

`CommentBody` играет ту же роль, но для комментария.

Он создается через:

- `CommentBody::new(&str) -> Result<CommentBody, DomainError>`

и читается через:

- `as_str()`

Смысл отдельного типа здесь тот же:

- комментарий и пост оба содержат текст;
- но на уровне модели это разные значения;
- их удобнее и безопаснее различать типами.

### `Timestamp`

`Timestamp` это доменный тип времени.

Его смысл в текущей реализации довольно простой:

- хранить unix seconds;
- не размазывать по коду сырые `u64`, когда речь идет именно о доменном времени.

Создается через:

- `Timestamp::from_unix_seconds(u64)`

Читается через:

- `as_unix_seconds()`

Пока у этого типа нет сложной валидации, но он уже задает ясную форму данных для всех верхних модулей.

## `Visibility`: что это за тип

`Visibility` это перечисление режимов видимости контента.

Сейчас наружу экспортируются варианты:

- `Public`
- `FriendsOnly`
- `MembersOnly`
- `PrivateCommunity`

Смысл этого типа:

- не хранить режим доступа как произвольную строку внутри доменной модели;
- явно перечислить известные режимы;
- дать верхним модулям типизированный способ выразить видимость поста или комментария.

Важно понимать ограничение:

- сам по себе `Visibility` еще не реализует полноценную политику авторизации и доставки;
- он задает форму значения, а не весь механизм применения правил доступа.

## Сущность `Post`: что она означает

`Post` это доменная сущность записи в ресурсе.

Она хранит:

- свой идентификатор;
- идентификатор ресурса;
- автора;
- время создания;
- текст;
- режим видимости;
- флаг скрытия.

### Как создается `Post`

Пост создается через:

- `Post::new(id, resource_id, author_id, created_at, body, visibility) -> Result<Post, DomainError>`

Смысл такого интерфейса:

- пост нельзя получить “как попало”;
- нужно явно передать уже валидные идентификаторы и body;
- конструктор сразу фиксирует исходное доменное состояние.

### Что можно читать у `Post`

У поста доступны методы:

- `id()`
- `resource_id()`
- `author_id()`
- `created_at()`
- `body()`
- `visibility()`
- `is_hidden()`

Снаружи это означает: верхний слой может безопасно читать доменные данные, но не должен произвольно менять поля напрямую.

### Что можно делать с `Post`

Сейчас у поста есть две доменные операции:

- `hide(self) -> Self`
- `edit(self, body: PostBody) -> Self`

Их смысл:

- `hide` переводит пост в скрытое состояние;
- `edit` заменяет текст поста на новый валидный `PostBody`.

Важно, что эти методы:

- не мутируют объект по ссылке;
- а возвращают новое значение.

То есть модель сейчас построена в более функциональном стиле: получил сущность, преобразовал ее, сохранил новый результат дальше.

## Сущность `Comment`: что она означает

`Comment` это доменная сущность комментария к посту.

Она хранит:

- собственный `comment_id`;
- `post_id`, к которому относится комментарий;
- опциональный `parent_comment_id`, если это ответ на другой комментарий;
- автора;
- время создания;
- текст;
- видимость;
- флаг скрытия.

### Как создается `Comment`

Комментарий создается через:

- `Comment::new(id, post_id, parent_comment_id, author_id, created_at, body, visibility) -> Result<Comment, DomainError>`

Смысл интерфейса здесь такой же, как у `Post`:

- построение комментария централизовано;
- невалидные body и id отсекаются до появления сущности;
- древовидность выражается явно через `parent_comment_id`.

### Что можно читать у `Comment`

У комментария доступны методы:

- `id()`
- `post_id()`
- `parent_comment_id()`
- `author_id()`
- `created_at()`
- `body()`
- `visibility()`
- `is_hidden()`

### Что можно делать с `Comment`

Сейчас есть две операции:

- `hide(self) -> Self`
- `edit(self, body: CommentBody) -> Self`

То есть внешний код может выразить базовые доменные переходы:

- скрыть комментарий;
- заменить текст комментария.

## Доменные события: зачем они нужны

Сущности описывают текущее состояние объекта, а доменные события описывают факт произошедшего изменения.

Для LiveLetters это особенно важно, потому что более верхние слои строят систему как event-driven contour: события дальше идут в протокол, outbox, sync и диагностику.

Сейчас модуль экспортирует такие типы событий:

- `PostCreated`
- `CommentCreated`
- `PostHidden`
- `CommentEdited`
- `CommentHidden`

Идея у всех одинаковая:

- событие хранит, что именно произошло;
- кто это сделал;
- к какому ресурсу и объекту это относится;
- когда это произошло;
- и, где нужно, какие дополнительные связи важны.

### `PostCreated`

Это событие факта “пост создан”.

Оно нужно верхним слоям, когда после создания поста нужно:

- зафиксировать доменное изменение;
- превратить его в protocol payload;
- положить его в outbox;
- потом реплицировать в другие клиенты.

Создается через `new(...)` и затем читается через геттеры:

- `event_id()`
- `post_id()`
- `resource_id()`
- `actor_id()`
- `created_at()`
- `visibility()`

### `CommentCreated`

Это событие факта “комментарий создан”.

Помимо обычных полей, оно важно тем, что хранит:

- `post_id`;
- `parent_comment_id`.

То есть оно явно выражает место комментария в дереве обсуждения.

### `PostHidden`

Это событие факта “пост скрыт”.

Оно нужно, когда систему интересует именно изменение видимости/состояния поста, а не полный снимок объекта.

### `CommentEdited`

Это событие факта “комментарий изменен”.

Его смысл в том, чтобы выразить прикладное изменение комментария как отдельный доменный факт.

### `CommentHidden`

Это событие факта “комментарий скрыт”.

Нужно по той же логике, что и `PostHidden`, только для комментария.

## Как устроен интерфейс событий

Все события создаются явным `new(...)`.

После создания с ними предполагается делать две вещи:

1. читать поля через геттеры;
2. передавать их в более верхние слои, где они уже превращаются в protocol payload, persist-артефакты или диагностические записи.

Снаружи важно понимать, что эти типы:

- сами не сериализуются в email-формат;
- сами не пишутся в БД;
- сами не проверяют transport-ограничения.

Они только задают доменную форму факта.

## `DomainError`: что означает эта ошибка

`DomainError` это внешний тип ошибок доменной валидации.

Сейчас он содержит только два варианта:

- `BlankIdentifier(&'static str)`
- `BlankBody(&'static str)`

Это значит, что на текущем этапе доменный модуль гарантирует хотя бы следующее:

- нельзя создать сущность или событие с пустыми идентификаторами;
- нельзя создать post/comment body из пустого текста.

Важно, что это не “техническая ошибка” и не “ошибка БД”.

`DomainError` означает:

- входные данные не соответствуют минимально допустимой форме предметной модели.

Для верхних слоев это важный сигнал: такую ошибку нужно трактовать как validation problem, а не как сбой инфраструктуры.

## Как верхние слои должны использовать этот модуль

Правильный сценарий использования выглядит так:

1. внешний слой получает сырые входные данные;
2. переводит их в доменные идентификаторы и value objects;
3. строит сущность или доменное событие через этот модуль;
4. только после этого передает результат дальше в protocol, store или app-core.

То есть `liveletters-domain` должен быть первым местом, где входные данные перестают быть “просто строками” и становятся осмысленной моделью.

## Что этот модуль специально не делает

Чтобы не было ложных ожиданий, важно зафиксировать и ограничения.

Этот модуль сейчас не делает следующего:

- не хранит сущности;
- не сериализует их;
- не знает про email и MIME;
- не знает про синхронизацию;
- не знает про права доступа как полноценную policy-систему;
- не строит дерево комментариев автоматически;
- не содержит полной модели аккаунтов, ресурсов, membership и subscription, хотя такие направления описаны в более общих архитектурных документах.

Иными словами, это минимальный, но реальный доменный API текущей реализации, а не окончательная полная модель LiveLetters.

## Что считать стабильным контрактом здесь

Если смотреть глазами потребителя crate, то наиболее значимыми частями интерфейса являются:

1. конструкторы `new(...)` у идентификаторов и value objects;
2. конструкторы `Post::new(...)` и `Comment::new(...)`;
3. доменные операции `hide` и `edit`;
4. перечисление `Visibility`;
5. набор и структура экспортируемых доменных событий;
6. `DomainError` как форма доменной validation-ошибки.

Если эти элементы меняются, это почти наверняка требует синхронного пересмотра:

- `liveletters-app-core`;
- `liveletters-protocol`;
- тестов, которые опираются на доменную модель.
