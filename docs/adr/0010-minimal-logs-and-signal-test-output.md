# ADR 0010: Минимально необходимые логи и сигнальный вывод тестов

## Статус

Принято.

## Дата

2026-04-09

## Суть

- В рабочем контуре допускаются только минимально необходимые логи.
- Временные отладочные логи должны быть краткоживущими и удаляться после устранения дефекта.
- Тесты по умолчанию запускаются в тихом режиме с акцентом на итоговый результат, а не на поток подробного вывода.
- Полный verbose-вывод и расширенное логирование включаются только на время диагностики.

## Контекст

В длительных сессиях избыточное логирование приводит к двум проблемам:

- ухудшается полезный сигнал в консоли и отчетах;
- растет шум в рабочем контексте и сложнее быстро выделять причины падений.

Похожая проблема возникает с тестовыми прогонами без ограничения вывода:

- важные ошибки теряются в большом потоке технических строк;
- ревью результатов и повторная диагностика занимают больше времени.

## Решение

### 1. Политика логов

- Логи добавляются только для конкретной диагностической цели.
- Каждый временный лог должен иметь понятный критерий удаления.
- Если лог нужен только для локальной отладки, он не должен становиться постоянной частью runtime.
- Временные runtime-логи пишутся в `.docs/runtime-logs` и не попадают в Git.

### 2. Политика вывода тестов

- Базовый запуск тестов должен давать компактный и сигнальный вывод.
- Подробный вывод включается только для расследования конкретного падения.
- После диагностики запуск возвращается к базовому тихому режиму.

Примеры базового режима:

- Rust: `cargo test -q`
- Clojure/ClojureScript: команды, дающие итоговую сводку без лишнего trace по умолчанию.

Примеры диагностического режима:

- Rust: запуск конкретного теста, `-- --nocapture` или `RUST_BACKTRACE=1`
- Clojure/ClojureScript: запуск конкретного namespace/теста и расширенный вывод только на время расследования.

## Последствия

Плюсы:

- меньше шумовых данных в консоли, отчетах и рабочем контексте;
- проще читать причины падений;
- быстрее локализуются регрессии;
- меньше риск оставить постоянный диагностический мусор в коде.

Минусы:

- нужно дисциплинированно удалять временные логи;
- иногда требуется дополнительный шаг для переключения в verbose-режим.

## Правила

1. Отладочные логи добавляются только под конкретную задачу диагностики.
2. Временные логи удаляются сразу после закрытия дефекта.
3. Локальные runtime-логи пишутся в `.docs/runtime-logs` и не коммитятся.
4. Тесты по умолчанию запускаются в компактном режиме.
5. Verbose-режим используется только временно для диагностики и затем отключается.

## Когда пересматривать

Решение нужно пересмотреть, если:

- текущие quiet-режимы тестов перестанут давать достаточный сигнал;
- появится централизованный формат логирования с автоматической политикой очистки;
- появятся обязательные требования CI к расширенному постоянному логированию.
