# ADR 0003: Обязательные команды проверки и общие служебные утилиты

## Статус

Принято.

## Дата

2026-04-08

## Суть

- Каждый подпроект обязан иметь команды линтинга, тестирования и форматирования.
- Для подпроекта желательно иметь одну агрегирующую команду полного прогона.
- Предпочтительный порядок в ней: линтинг, тесты, форматирование.
- Общую служебную логику по возможности нужно выносить в корневую папку `utils`.

## Контекст

Если у подпроектов разные и несогласованные команды проверки, monorepo быстро становится неудобным:

- трудно помнить команды;
- трудно автоматизировать локальную работу;
- в подпроектах начинает копироваться одна и та же служебная логика.

Есть явное пожелание иметь одну удобную команду, которая запускает все основные проверки подряд.

## Решение

Каждый подпроект должен предоставлять команды:

- линтинга;
- тестирования;
- форматирования.

Желательно также иметь одну агрегирующую команду, которая запускает их последовательно:

1. линтинг;
2. тесты;
3. форматирование.

Это решение принято осознанно ради удобства локальной работы, несмотря на известные минусы смешения проверяющих и изменяющих операций в одной последовательности.

Общая служебная логика не должна без нужды копироваться в каждом подпроекте. Для нее используется корневая папка `utils`.

Подпроект может иметь локальный запускатель, но он по возможности должен быть тонким и опираться на `utils`.

## Отдельное правило для ClojureScript

## Последствия

Плюсы:

- единообразные команды по всему репозиторию;
- удобнее локальная разработка;
- меньше дублирования служебных скриптов;
- проще развивать общие запускатели.

Минусы:

- единая команда не всегда оптимальна для всех сценариев;
- `utils` требует дисциплины, иначе превратится в склад случайных скриптов.

## Правила

1. Каждый подпроект обязан иметь команды линтинга, тестирования и форматирования.
2. Для каждого подпроекта желательно иметь одну агрегирующую команду.
3. Предпочтительный порядок в агрегирующей команде: линтинг, тесты, форматирование.
4. Общая служебная логика по возможности выносится в `utils`.
5. Каждый подпроект обязан документировать свои команды проверки.

## Когда пересматривать

Решение надо пересмотреть, если:

- единая агрегирующая команда системно мешает CI или локальной работе;
- `utils` слишком разрастается;
- разные технологические стеки начинают требовать несовместимых подходов.
