# ADR 0002: Инварианты разработки и внутренние документы подпроектов

## Статус

Принято.

## Дата

2026-04-08

## Суть

- Если это осмысленно, реализация или исправление начинается с красных тестов.
- Документация о реализованном поведении пишется или обновляется только после зеленых тестов.
- Основной язык документации это русский язык без лишних англицизмов.
- В публичных документах нельзя использовать абсолютные пути файловой системы.
- Если в документ добавляется ссылка на файл или документ, она должна быть сразу кликабельной.
- Каждый подпроект может и должен иметь скрытые каталоги `.plans` и `.docs`.
- Внутренние документы именуются с временным префиксом `YYMMDD-HHMMSS-...`.

## Контекст

Monorepo с несколькими подпроектами быстро теряет единообразие, если:

- тесты пишутся в разном порядке;
- документация обновляется случайно;
- в публичных документах появляются машинно-зависимые абсолютные пути;
- связанные документы перечисляются как обычный текст без кликабельных ссылок;
- внутренние рабочие документы хранятся хаотично;
- язык документации не нормирован.

Нужны общие инварианты, а не неявные привычки.

## Решение

### Тесты

Если поведение можно выразить тестом, предпочтительный порядок такой:

1. написать тест;
2. увидеть красное состояние;
3. реализовать изменение;
4. получить зеленые тесты;
5. только после этого обновить документацию.

Фраза "если это осмысленно" означает, что правило не применяется механически к каркасной или исследовательской работе.

### Документация

Документация, описывающая текущее реализованное поведение, должна обновляться только после того, как реализация подтверждена зелеными тестами.

Публичные документы не должны содержать абсолютные пути вроде `/home/user/...`, так как такие пути зависят от конкретной машины и засоряют репозиторий.

Если документ ссылается на другой файл или документ внутри репозитория, ссылка должна быть оформлена как кликабельная markdown-ссылка, а не как простой текстовый путь.

### Язык

Документация пишется по-русски.

Английские термины допустимы только там, где без них теряется точность или ломается смысл.

### Внутренние документы

Для подпроектов используются скрытые каталоги:

- `.plans` для планов;
- `.docs` для прочих внутренних документов.

Эти каталоги должны быть скрыты от Git по умолчанию.

Имена файлов начинаются с временного префикса:

- `YYMMDD-HHMMSS-<short-name>.md`

Время в имени должно примерно совпадать с реальным временем создания документа.

## Последствия

Плюсы:

- единый порядок реализации;
- меньше рассинхрона между кодом и документацией;
- публичные документы становятся переносимыми между машинами;
- связанные материалы легче открывать и проверять;
- рабочие документы становятся лучше организованы;
- тексты проекта становятся более цельными по языку.

Минусы:

- потребуется дисциплина;
- timestamp-именование создает небольшое ручное усилие;
- правило про тесты нельзя применять механически без инженерного здравого смысла.

## Правила

1. Для реализуемого поведения, если это осмысленно, сначала пишутся красные тесты.
2. Документация о реализованном поведении обновляется только после зеленых тестов.
3. Основной язык документации это русский язык.
4. В публичных документах запрещены абсолютные пути файловой системы.
5. Ссылки на файлы и документы в документации должны быть кликабельными.
6. Подпроекты могут и должны использовать `.plans` и `.docs`.
7. Внутренние документы получают временной префикс `YYMMDD-HHMMSS-...`.
8. `.plans` и `.docs` исключаются из Git по умолчанию.

## Когда пересматривать

Решение надо пересмотреть, если:

- правило про красные тесты системно мешает значимому классу задач;
- появится более удачный формат для внутренних документов;
- правило про кликабельные ссылки окажется неприменимым для значимого класса документов;
- языковое правило окажется слишком жестким для части технических материалов.
