# ADR 0009: Консервативный frontend app-store на `atom`, без обязательного `re-frame`

## Статус

Принято.

## Дата

2026-04-09

## Суть

- Для frontend LiveLetters принимается консервативная state-centric модель.
- Базовый ожидаемый UI-путь это `Reagent`.
- На frontend должен существовать единый `app-store` как центр прикладного состояния и orchestration.
- Базовая техническая форма этого store это один корневой `atom` или малое число явно обоснованных `atom`.
- `liveletters-ui-model` отвечает за чистые selectors, форматирование и view model.
- `liveletters-frontend-api` отвечает за вызовы backend и подписки на backend events.
- `liveletters-frontend-app` отвечает за организацию app-store, intents и orchestration поверх этих слоев.
- `re-frame` не считается обязательным базовым слоем и не вводится по умолчанию.

## Контекст

Для ClojureScript-приложения можно выбрать более событийную и каркасную модель состояния, например через `re-frame`, либо более прямую state-centric модель.

В обсуждении frontend-части зафиксировано предпочтение к удобной работе с многосущностным состоянием, близкой по практическому ощущению к тому, за что в React/TypeScript-проектах часто выбирают `MobX`:

- состояние удобно читать;
- состояние удобно менять;
- производные представления выражаются как обычные функции;
- нужен сильный прикладной слой на клиенте, который по роли похож на “второй backend”;
- но не хочется по умолчанию вводить тяжелый event-centric каркас, если целевая модель ближе к state-centric/reactive подходу.

При этом есть и общий стиль проекта:

- консервативный выбор технологий;
- минимум лишних слоев;
- явные границы между backend API, view model и приложением.

## Решение

Для frontend LiveLetters принимается следующая схема.

### 1. UI-слой

Ожидаемая базовая UI-библиотека:

- `Reagent`

Это консервативный и устойчивый путь для ClojureScript UI, который хорошо сочетается с моделью состояния на `atom`.

### 2. App-store

Базовая схема frontend state:

- единый `app-store`
- с базовой технической формой в одном корневом `atom`

Этот store должен играть роль клиентского application layer:

- хранить прикладное состояние;
- координировать intents;
- скрывать backend integration;
- собирать производные screen-friendly данные вместе с `liveletters-ui-model`;
- разгружать экранный слой.

Допустимо использовать более одного `atom`, но только когда это действительно оправдано отдельной ответственностью, а не случайным ростом кода.

Предпочтение отдается одной ясной схеме store и состояния, а не разрозненному набору независимых атомов без общей структуры.

При этом это решение не означает, что вообще все состояние без исключения нужно вытаскивать в глобальный store.

В `app-store` должно попадать прежде всего:

- прикладное состояние;
- межэкранные сценарии;
- данные, загруженные с backend;
- sync и diagnostics;
- кэш сущностей;
- важные формы и transitions, влияющие на несколько частей приложения.

Локальное эфемерное UI-состояние допустимо и разумно держать рядом с компонентом, если оно:

- не влияет на другие части приложения;
- не является частью прикладного сценария;
- не требует координации через app-store.

### 3. Разделение ответственности

- `liveletters-frontend-api` вызывает backend и подписывается на backend events;
- `liveletters-ui-model` преобразует DTO и state slices в view model;
- `liveletters-frontend-app` организует `app-store`, intents, effects и orchestration;
- `liveletters-ui-kit` предоставляет переиспользуемые компоненты и primitives.

### 4. Производные данные

Производные данные и screen-friendly представления должны выражаться:

- через чистые функции;
- через selectors;
- через view model builders.

Эта логика не должна смешиваться с вызовами backend или с низкоуровневыми UI-компонентами.

### 5. Что не принимается по умолчанию

По умолчанию не принимается:

- обязательное введение `re-frame` как базового слоя;
- ранний переход к event-heavy архитектуре на frontend как к обязательной модели всего приложения;
- смешивание backend integration и глобального UI state в одном модуле;
- ошибочная трактовка, что любой локальный UI toggle или transient state обязан жить в app-store.

`re-frame` может быть введен позже только при реальной доказанной необходимости, а не как стартовая архитектурная установка.

## Последствия

Плюсы:

- на frontend появляется явный клиентский application layer;
- состояние фронтенда остается понятным и обозримым;
- удобно строить многосущностные экраны и сценарии;
- хорошо сочетается с уже выделенными `frontend-api` и `ui-model`;
- легче разгрузить менее квалифицированный UI-слой;
- легче двигаться малыми проходами, не вводя тяжелый event-centric каркас заранее.

Минусы:

- потребуется дисциплина в структуре app-store;
- без аккуратных selectors, intents и модульного разбиения один `atom` можно превратить в беспорядочную карту;
- если приложение сильно вырастет, позже может понадобиться дополнительный state layer.

## Правила

1. Базовый UI-путь для frontend это `Reagent`.
2. На frontend должен существовать единый `app-store` как центр прикладного состояния.
3. Базовая техническая форма этого store по умолчанию это один корневой `atom`.
4. Дополнительные `atom` допустимы только при явной ответственности.
5. Не все вообще состояние обязано жить в app-store.
6. Локальное эфемерное UI-состояние допустимо держать рядом с компонентом.
7. `liveletters-ui-model` должен оставаться слоем чистых функций и selectors.
8. `liveletters-frontend-api` не должен становиться хранилищем UI-состояния.
9. `liveletters-frontend-app` отвечает за app-store, intents, effects и orchestration.
10. `re-frame` не вводится по умолчанию и требует отдельного осознанного решения.

## Влияние на уже пройденные этапы

На текущий момент это решение не требует отката или серьезной переделки уже выполненных этапов.

Совместимость по слоям такая:

- `liveletters-frontend-api` уже совместим, потому что он выделен как отдельный boundary вызовов и событий;
- `liveletters-ui-model` уже совместим, потому что он собран как набор чистых функций;
- `liveletters-rust-backend-app` уже совместим, потому что не навязывает frontend state framework;
- следующий значимый слой, где это решение должно проявиться явно, это `liveletters-frontend-app` как app-store слой.

То есть сейчас речь идет не о переработке уже сделанного, а о том, чтобы не увести этапы 11-12 в иную архитектуру.

## Когда пересматривать

Решение надо пересмотреть, если:

- frontend state станет слишком сложным для текущей формы app-store;
- появится доказанная необходимость в event-centric каркасе;
- `Reagent` окажется плохим выбором для реального UI-контура проекта;
- появятся требования, которые удобнее решать иным state layer.
