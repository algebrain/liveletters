# ADR 0011: Подпроекты с внешним программным интерфейсом обязаны иметь `INTERFACE.md`

## Статус

Принято.

## Дата

2026-04-09

## Суть

- Если подпроект предоставляет внешний программный интерфейс, у него обязан быть файл `INTERFACE.md`.
- Создание такого подпроекта требует одновременного создания `INTERFACE.md`.
- Изменение внешнего программного интерфейса такого подпроекта требует одновременного обновления `INTERFACE.md`.
- `INTERFACE.md` должен описывать не только перечень имен, но и смысл интерфейса:
  - зачем нужны экспортируемые типы и функции;
  - что с ними можно делать;
  - какие входы, выходы, ограничения и гарантии считаются частью контракта.

## Контекст

В репозитории появились подпроекты, которые уже предоставляют внешние программные интерфейсы:

- Rust crates;
- backend app boundary;
- frontend integration и presentation модули.

Без отдельного правила такие интерфейсы легко начинают жить в одном из двух плохих режимов:

1. интерфейс существует только в коде и его приходится каждый раз заново читать по исходникам;
2. рядом лежит формальный документ, который быстро вырождается в список имен без объяснения смысла и практического использования.

Для архитектурно значимых подпроектов этого недостаточно.

Нужен отдельный документ, который:

- фиксирует внешний программный интерфейс;
- объясняет его по-русски;
- помогает понять границы ответственности;
- позволяет читать модуль сверху вниз, не начиная сразу с исходников.

## Решение

Для подпроектов, предоставляющих внешний программный интерфейс, вводится обязательный документ:

- `INTERFACE.md`

Под "внешним программным интерфейсом" здесь понимается интерфейс, которым реально пользуется другой код за пределами самого подпроекта, например:

- публичный Rust API crate;
- backend command/event boundary;
- frontend integration boundary;
- frontend view-model boundary;
- библиотека переиспользуемых UI-компонентов.

Правило применения:

1. Если создается новый подпроект с внешним программным интерфейсом, вместе с ним создается `INTERFACE.md`.
2. Если меняется внешний программный интерфейс существующего подпроекта, вместе с кодом обновляется `INTERFACE.md`.
3. Если меняется только внутренняя реализация без изменения внешнего интерфейса, обязательного обновления `INTERFACE.md` нет.

## Что должен содержать `INTERFACE.md`

Минимально документ должен объяснять:

- назначение подпроекта как внешней границы;
- где находится интерфейс;
- что считается внешним интерфейсом подпроекта;
- смысл основных экспортируемых типов, функций, команд, событий или компонентов;
- как верхние слои должны использовать этот интерфейс;
- что модуль специально не делает;
- что считается наиболее стабильной частью контракта.

Важно:

- `INTERFACE.md` не должен вырождаться в голый список имен;
- документ должен объяснять интерфейс человеческим языком;
- документ должен описывать реальный текущий интерфейс по коду, а не абстрактный "идеальный" будущий API.

## Последствия

Плюсы:

- внешний API подпроектов становится явным и читаемым;
- проще заходить в код сверху вниз;
- легче замечать архитектурные изменения интерфейса;
- проще согласовывать backend, frontend и библиотечные boundary между собой.

Минусы:

- появляется дополнительная дисциплина;
- при изменении публичного API нужно не забывать обновлять документацию;
- плохой `INTERFACE.md` может создать ложное ощущение документированности, если его поддерживать формально.

## Правила

1. Подпроект с внешним программным интерфейсом обязан иметь `INTERFACE.md`.
2. Создание такого подпроекта требует создания `INTERFACE.md`.
3. Изменение внешнего программного интерфейса требует обновления `INTERFACE.md`.
4. `INTERFACE.md` описывает реальный текущий интерфейс, а не только желаемый будущий.
5. `INTERFACE.md` должен объяснять смысл интерфейса, а не ограничиваться перечнем экспортов.

## Когда пересматривать

Решение надо пересмотреть, если:

- окажется, что часть подпроектов удобнее документировать в ином формате;
- появится более строгая общая схема API-документации для всего репозитория;
- правило окажется слишком широким или слишком размытым для реального применения.
