# ADR 0007: Не складывать подпроект в один `.rs` или `.cljs`

## Статус

Принято.

## Дата

2026-04-08

## Суть

- Подпроект не должен жить в одном большом `.rs`, `.clj`, `.cljs` или `.cljc` файле.
- Код нужно разносить по тематическим файлам и модулям.
- Разбиение должно отражать ответственность, а не случайный рост файла.

## Контекст

На раннем этапе удобно быстро начать с одного файла. Но по мере роста подпроекта это быстро приводит к проблемам:

- код трудно читать;
- труднее искать границы ответственности;
- растет риск случайных связей;
- тяжелее тестировать и расширять код;
- труднее обсуждать и ревьюить изменения.

В `liveletters-domain` это уже проявилось практически: после первых шагов этапа 1 один `lib.rs` пришлось разносить по тематическим частям.

## Решение

Для подпроектов LiveLetters принимается правило:

- не держать весь значимый код подпроекта в одном файле;
- как только появляются устойчивые смысловые части, их нужно выносить в отдельные файлы;
- структура файлов должна следовать за доменной и технической декомпозицией.

Типичные оси разбиения:

- ошибки;
- идентификаторы;
- value objects;
- сущности;
- события;
- transport adapters;
- queries;
- commands;
- UI components;
- view model;
- integration layer.

## Что правило не означает

Это не означает, что:

- любой маленький модуль нужно дробить искусственно;
- нужно создавать много пустых файлов;
- разбиение должно быть максимально мелким.

Смысл правила в другом:

- не допускать, чтобы один файл становился фактическим контейнером всего подпроекта.

## Последствия

Плюсы:

- проще читать код;
- проще поддерживать границы ответственности;
- проще тестировать и рефакторить;
- проще делать ревью.

Минусы:

- чуть больше файлов и навигации;
- нужна дисциплина, чтобы разбиение оставалось осмысленным.

## Правила

1. Подпроект не должен развиваться как один большой исходный файл.
2. Код нужно разносить по тематическим файлам и модулям.
3. Разбиение должно соответствовать ответственности, а не случайной истории роста.
4. Искусственное дробление без смысла тоже нежелательно.

## Когда пересматривать

Решение надо пересмотреть, если:

- для какого-то класса маленьких подпроектов правило окажется избыточным;
- появится более удачное и точное правило структурирования кода.
