# ADR Index

## Назначение

Этот каталог хранит архитектурные решения проекта.

Правило чтения:

- сначала читать этот индекс;
- полный ADR открывать только если нужна мотивация или точные последствия.


## 0001 Monorepo из независимых подпроектов и pinning по commit hash

Файл:

- [0001-monorepo-and-version-pinning.md](./0001-monorepo-and-version-pinning.md)

Суть:

- проект хранится в одном monorepo;
- приложения живут в `apps/`, библиотеки в `modules/`;
- общего workspace в корне нет;
- каждый подпроект живет как отдельный проект;
- commit hash считается допустимым способом фиксации версии.

Главное последствие:

- подпроекты изолированы сильнее и зависят друг от друга явно.


## 0002 Инварианты разработки и внутренние документы

Файл:

- [0002-development-invariants-and-internal-documents.md](./0002-development-invariants-and-internal-documents.md)

Суть:

- если это осмысленно, работу начинаем с красных тестов;
- документацию о реализованном поведении меняем только после зеленых тестов;
- документацию пишем по-русски без лишних англицизмов;
- в публичных документах не используем абсолютные пути;
- ссылки на документы и файлы оформляем кликабельно;
- у каждого подпроекта могут быть скрытые каталоги `.plans` и `.docs`;
- внутренние документы именуются с временным префиксом `YYMMDD-HHMMSS-...`.

Главное последствие:

- процесс разработки и рабочие документы должны быть единообразны по всему репозиторию.


## 0003 Команды проверки и общие утилиты

Файл:

- [0003-test-lint-format-and-shared-utils.md](./0003-test-lint-format-and-shared-utils.md)

Суть:

- каждый подпроект обязан иметь команды линтинга, тестов и форматирования;
- желательно иметь одну агрегирующую команду;
- общую служебную логику по возможности нужно выносить в корневую `utils`.

Главное последствие:

- проверки должны быть единообразны, а служебные запускатели не должны бесконтрольно дублироваться.


## 0004 Не использовать локальный `.m2`

Файл:

- [0004-no-local-m2.md](./0004-no-local-m2.md)

Суть:

- внутри репозитория нельзя делать локальный `.m2`;
- нужно использовать глобальное Maven-хранилище, настроенное в среде.

Главное последствие:

- проблемы зависимостей нужно чинить через конфигурацию проекта или среды, а не локальным обходом.


## 0005 Инкрементальные отчеты в `.docs`

Файл:

- [0005-incremental-reports-in-docs.md](./0005-incremental-reports-in-docs.md)

Суть:

- по итогам значимой сессии или этапа нужно вести инкрементальный отчет;
- отчет хранится в `.docs`;
- отчет может быть глобальным или локальным для подпроекта.

Главное последствие:

- хронология работы и фактический прогресс должны фиксироваться системно, а не оставаться только в истории чата.


## 0006 Заморозить `idea.preliminary.md` и `idea.technical.md`

Файл:

- [0006-freeze-idea-documents.md](./0006-freeze-idea-documents.md)

Суть:

- два документа идеи считаются историческими;
- они больше не редактируются в обычном порядке;
- это правило относится только к ним и не означает, что на каждое изменение нужен новый документ.

Главное последствие:

- исходные формулировки идеи сохраняются как исторический слой проекта.


## 0007 Не складывать подпроект в один `.rs` или `.cljs`

Файл:

- [0007-no-single-file-subprojects.md](./0007-no-single-file-subprojects.md)

Суть:

- подпроект не должен жить в одном большом исходном файле;
- код нужно разносить по тематическим файлам и модулям;
- разбиение должно отражать ответственность.

Главное последствие:

- структура кода должна расти вместе с архитектурой, а не оставаться одним большим файлом.


## 0008 В тестах использовать временный `HOME`

Файл:

- [0008-temporary-home-in-tests.md](./0008-temporary-home-in-tests.md)

Суть:

- тесты не должны зависеть от реального домашнего каталога разработчика;
- если код использует `HOME` или домашнюю директорию по умолчанию, тест обязан подменять ее временным каталогом;
- тестовые данные не должны попадать в настоящий `~/.liveletters`.

Главное последствие:

- любые тесты хранения и путей должны быть изолированы от реальной пользовательской среды.
