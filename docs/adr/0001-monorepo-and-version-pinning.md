# ADR 0001: Monorepo из независимых подпроектов и pinning версий по commit hash

## Статус

Принято.

## Дата

2026-04-08

## Суть

- LiveLetters хранится в одном monorepo.
- Приложения живут в `apps/`, библиотеки и прочие модули в `modules/`.
- Общего Rust workspace в корне нет.
- Общего ClojureScript workspace в корне нет.
- Каждый подпроект живет как отдельный проект со своей конфигурацией.
- Commit hash считается допустимым способом pinning версии.

## Контекст

Проект состоит из нескольких приложений и библиотек. Нужна структура, которая:

- не требует отдельного репозитория на каждый модуль;
- позволяет держать подпроекты в одном репозитории, но считать их независимыми;
- сохраняет возможность подключать модули по Git на конкретный commit;
- минимизирует неявную связанность между подпроектами.

## Решение

Используется один Git-репозиторий.

В нем:

- верхнеуровневые приложения живут в `apps/`;
- библиотеки и прочие модули живут в `modules/`;
- каждый подпроект имеет собственные конфигурационные файлы;
- подпроекты не зависят от общего workspace в корне;
- commit hash можно использовать как фиксированную версию зависимости.

При этом для ранних этапов сквозной реализации принимается практическое переходное правило:

- внутри текущего monorepo допустимы локальные зависимости между подпроектами;
- для Rust это `path`-зависимости;
- для ClojureScript это локальные `:paths` и `:source-paths`;
- `git + rev` не считается обязательным стартовым режимом для внутренней разработки на раннем этапе.

Иными словами:

- архитектурно подпроекты считаются независимыми;
- practically на раннем этапе их допустимо связывать локально;
- переход к `git + rev` ожидается позже, когда API модулей стабилизируются и появится смысл проверять их как более самостоятельные единицы.

## Технические детали для Rust

Rust-подпроекты живут как независимые cargo-проекты.

Если один подпроект зависит от другого, целевая модель такая:

- зависимость подключается как отдельный Git dependency;
- версия фиксируется по commit hash;
- подпроект не полагается на общий `Cargo workspace`.

Но на раннем этапе текущая рабочая схема может быть проще:

- зависимость подключается через локальный `path`;
- это считается допустимым временным рабочим режимом для сквозной реализации;
- перевод на `git` dependency с `rev` делается позже, когда модульная граница уже достаточно стабилизирована.

Внешний пример зависимости:

```toml
[dependencies]
liveletters-domain = { git = "https://github.com/your-org/liveletters.git", rev = "abc123...", package = "liveletters-domain" }
```

Это означает, что monorepo используется как физическое хранилище кода, но не как общий Rust workspace.

## Технические детали для ClojureScript

Для ClojureScript подпроекты тоже считаются независимыми.

На практике это означает:

- у frontend app свои `deps.edn` и `shadow-cljs.edn`;
- зависимые модули подключаются явно;
- commit hash остается допустимым способом фиксировать версии;
- отсутствие общего workspace приводит к большему количеству ручной конфигурации, но это считается приемлемой ценой за изоляцию подпроектов.

На раннем этапе допустимая рабочая схема такая:

- frontend app явно подключает соседние модули через локальные `:paths` и `:source-paths`;
- это не отменяет идею независимых подпроектов, а только упрощает раннюю сквозную разработку;
- более поздний переход к более изолированной схеме допустим после стабилизации границ и API.

## Последствия

Плюсы:

- меньше ранней организационной сложности;
- подпроекты изолированы сильнее;
- проще мыслить модули как самостоятельные проекты;
- легче переходить к схеме "каждый модуль почти как отдельный репозиторий";
- commit pinning остается естественным способом фиксации версий.
- локальные зависимости упрощают ранний вертикальный проход через несколько модулей.

Минусы:

- больше ручной настройки;
- меньше удобств общего workspace;
- для ClojureScript чаще встречаются шероховатости конфигурации.
- при слишком долгом сохранении локальных зависимостей можно отложить реальную проверку модульной изоляции.

## Правила

1. Каждый модуль живет в собственной подпапке.
2. Приложения живут в `apps/`.
3. Библиотеки и прочие модули живут в `modules/`.
4. Общего workspace в корне нет.
5. Каждый подпроект имеет собственные конфигурационные файлы.
6. Commit hash допускается как способ фиксировать версию.
7. На раннем этапе внутри monorepo допустимы локальные зависимости между подпроектами.
8. Переход на `git + rev` делается осознанно, когда модульная граница уже достаточно стабилизировалась.
9. Отдельные репозитории создаются только при реальной необходимости.

## Когда пересматривать

Решение надо пересмотреть, если:

- появятся независимые release cycle;
- появятся внешние потребители ClojureScript-модулей как отдельных библиотек;
- monorepo начнет мешать CI или выпуску версий.
