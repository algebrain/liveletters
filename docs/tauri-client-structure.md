## Примерная структура Tauri-клиента LiveLetters

### 1. Назначение документа

Этот документ описывает примерную внутреннюю структуру desktop-клиента LiveLetters на Tauri.

Цель структуры:

- разделить UI, доменную логику и почтовый транспорт;
- не превратить приложение в "почтовый клиент";
- дать понятные границы между frontend и Rust backend;
- подготовить архитектуру к постепенному росту от MVP к более сложной системе.


### 2. Общий принцип

Tauri-клиент разумно строить как приложение из трех основных слоев:

- frontend UI;
- application/backend layer;
- persistence and transport layer.

В терминах ответственности это выглядит так:

- frontend показывает блоги, посты, комментарии, подписки и настройки;
- backend исполняет команды, применяет доменные правила и координирует синхронизацию;
- transport и storage работают с IMAP/SMTP, локальной БД и файловыми артефактами.

Главный принцип: frontend не должен знать детали MIME, IMAP, SMTP, дедупликации и парсинга писем.


### 3. Логическая схема приложения

Удобно мыслить клиент как набор подсистем:

- `ui shell`;
- `application services`;
- `domain model`;
- `sync engine`;
- `mail gateway`;
- `local database`;
- `background jobs`;
- `diagnostics`.

Поток данных в общем виде:

1. Пользователь совершает действие в UI.
2. Frontend вызывает команду backend.
3. Backend создает доменное событие.
4. Доменное событие сохраняется в локальной БД.
5. При необходимости backend строит протокольное сообщение.
6. Mail gateway отправляет это сообщение по email.
7. Фоновые задачи получают входящие письма.
8. Sync engine извлекает из них доменные события, валидирует и применяет к локальной БД.
9. Frontend получает обновленное состояние через query-команды или события приложения.


### 4. Что должно жить во frontend

Frontend должен отвечать только за пользовательский опыт и представление данных.

Типичный состав frontend-части:

- навигация;
- экран ленты;
- экран блога;
- экран поста и треда комментариев;
- экран подписок;
- экран друзей;
- экран ресурсов пользователя;
- экран настроек аккаунта и почтовых подключений;
- экран диагностики и синхронизации;
- формы создания и редактирования постов и комментариев.

Во frontend желательно держать:

- маршрутизацию;
- состояние текущего экрана;
- кэш уже загруженных view model;
- optimistic UI там, где это безопасно;
- пользовательские фильтры, сортировки и локальные настройки отображения.

Во frontend нежелательно держать:

- правила авторизации доменных действий;
- парсинг протокольных сообщений;
- работу с IMAP и SMTP;
- логику дедупликации и anti-replay;
- прямой доступ к сырой локальной БД.


### 5. Что должно жить в Tauri backend

Rust backend должен быть основным носителем прикладной логики.

Именно backend должен отвечать за:

- выполнение команд пользователя;
- создание и применение доменных событий;
- валидацию входящих данных;
- работу с локальной БД;
- преобразование доменного события в протокольное сообщение;
- синхронизацию входящей и исходящей почты;
- фоновые задачи;
- диагностику и журналирование.

Такой подход дает:

- единый источник доменных правил;
- меньше дублирования логики между платформами;
- более безопасную работу с данными и учетными записями;
- возможность позже переиспользовать значительную часть backend-кода для mobile или CLI-компонентов.


### 6. Предлагаемое разбиение Rust backend на модули

Ниже не единственно возможная, но здоровая структура.

#### 6.1. `app`

Верхний уровень инициализации приложения:

- запуск Tauri;
- сборка зависимостей;
- регистрация команд;
- wiring сервисов;
- запуск фоновых задач;
- централизованная конфигурация.

#### 6.2. `config`

Модуль конфигурации:

- путь к данным;
- профиль пользователя;
- настройки логирования;
- параметры фоновой синхронизации;
- feature flags;
- настройки почтовых аккаунтов.

#### 6.3. `domain`

Чистая предметная модель:

- сущности;
- value objects;
- enums;
- инварианты;
- типы доменных событий;
- политики видимости и доступа.

Этот слой должен быть максимально независимым от Tauri, IMAP, SMTP и SQLite.

#### 6.4. `application`

Слой use cases и application services:

- создать пост;
- отредактировать пост;
- создать комментарий;
- подписаться на ресурс;
- импортировать входящие сообщения;
- переиндексировать локальную историю;
- получить ленту;
- получить тред обсуждения;
- получить статус синхронизации.

Именно здесь удобно держать orchestration:

- какие репозитории вызвать;
- какие доменные события сформировать;
- какие сообщения поставить в outbox;
- какие побочные задачи запустить.

#### 6.5. `protocol`

Описание межклиентского протокола:

- структура протокольного сообщения;
- JSON-схема payload;
- MIME-представление;
- заголовки `X-...`;
- сериализация и десериализация;
- версионирование схемы.

Это важный модуль, потому что он отделяет доменную модель от транспорта.

#### 6.6. `mail`

Шлюз почтового транспорта:

- SMTP-отправка;
- IMAP-получение;
- разбор папок;
- отметки уже обработанных сообщений;
- повторные попытки отправки;
- преобразование сырых email в внутреннюю форму.

Это транспортный слой. Он не должен сам принимать доменные решения.

#### 6.7. `sync`

Механизм синхронизации и репликации:

- импорт новых сообщений из IMAP;
- извлечение доменных событий;
- валидация;
- дедупликация;
- anti-replay;
- применение событий к локальному состоянию;
- обработка отложенных зависимостей;
- обновление materialized views.

По сути это один из центральных модулей системы.

#### 6.8. `storage`

Доступ к локальным данным:

- подключение к SQLite;
- миграции;
- репозитории;
- outbox;
- inbox journal;
- таблицы индексов;
- транзакции.

Если архитектура будет аккуратной, `storage` не должен знать UI, а `domain` не должен знать SQL.

#### 6.9. `jobs`

Фоновые задачи:

- периодическая синхронизация;
- отправка накопившегося outbox;
- переобработка отложенных событий;
- очистка временных данных;
- пересборка индексов;
- контроль "подвисших" сообщений.

#### 6.10. `api`

Граница между frontend и backend:

- Tauri commands;
- события для frontend;
- DTO и view models;
- типы ошибок, пригодные для UI.

Это важно выделить отдельно, чтобы команда UI не зависела от внутренней структуры backend-модулей.

#### 6.11. `diagnostics`

Диагностика и отладка:

- журнал входящих протокольных сообщений;
- журнал исходящих сообщений;
- ошибки парсинга;
- ошибки валидации;
- трассировка применения событий;
- экран технического состояния.

### 8. Граница между командами и запросами

У клиента почти наверняка будет полезна схема, близкая к CQRS-lite:

- команды меняют состояние;
- запросы читают готовое состояние.

Примеры команд:

- `create_post`;
- `edit_post`;
- `create_comment`;
- `subscribe_to_resource`;
- `connect_mail_account`;
- `run_sync_now`;
- `rebuild_indexes`.

Примеры запросов:

- `get_home_feed`;
- `get_resource_page`;
- `get_post_thread`;
- `get_account_settings`;
- `get_sync_status`;
- `list_incoming_failures`.

Это удобно, потому что:

- UI получает уже собранные view model;
- domain/application слой не подстраивается под конкретные компоненты интерфейса;
- проще управлять производительностью и кешированием.


### 9. Локальная БД и materialized views

Для LiveLetters клиент почти наверняка должен хранить не только сырые сообщения, но и вычисленное текущее состояние.

Практично разделить данные на несколько групп:

- канонические локальные сущности;
- журнал входящих протокольных сообщений;
- журнал исходящих протокольных сообщений;
- извлеченные доменные события;
- таблицы materialized views для UI;
- технические индексы синхронизации.

Условно это может выглядеть так:

- `accounts`;
- `resources`;
- `posts`;
- `comments`;
- `subscriptions`;
- `memberships`;
- `raw_messages`;
- `raw_events`;
- `outbox_messages`;
- `sync_cursors`;
- `failed_ingest`;
- `feed_items`;
- `thread_views`.

Важно, чтобы UI в основном читал не сырые события, а уже подготовленное материализованное состояние.


### 10. Работа с outbox и inbox

В клиенте удобно явно выделить два потока:

- `outbox`;
- `inbox ingest`.

`outbox` отвечает за:

- формирование протокольного сообщения;
- постановку в очередь отправки;
- попытки отправки;
- фиксацию статуса доставки;
- повторные попытки после ошибок.

`inbox ingest` отвечает за:

- чтение входящих писем;
- выделение LiveLetters-сообщений;
- классификацию;
- разбор payload;
- валидацию;
- применение доменных событий.

Такое разделение делает систему понятнее и устойчивее к сбоям.


### 11. Реактивность UI

Есть два нормальных варианта связи frontend и backend:

- frontend сам периодически делает query-запросы;
- backend дополнительно пушит события об изменении состояния.

Для MVP достаточно простой модели:

- команды вызываются через Tauri commands;
- после успешной команды UI инвалидирует свои запросы;
- синхронизация публикует ограниченное число app events, например "лента обновилась" или "появились ошибки импорта".

Нет смысла делать слишком сложную реактивную шину с самого начала.


### 12. Многопоточность и фоновые операции

Нужно изначально считать, что операции синхронизации и индексации могут быть долгими.

Поэтому:

- UI-поток не должен ждать IMAP или SMTP напрямую;
- длительные задачи должны жить в backend;
- нужна очередь фоновых работ;
- прогресс задач должен быть доступен UI;
- отмена и повторный запуск синхронизации должны быть контролируемыми.

Особенно важно не смешивать:

- пользовательское редактирование поста;
- отправку письма;
- фоновую обработку входящей истории.


### 13. Безопасность и секреты

Tauri-клиент будет работать с чувствительными данными:

- учетные данные почтовых аккаунтов;
- локальная история сообщений;
- возможно токены внешних сервисов;
- потенциально чувствительный контент закрытых записей.

Поэтому стоит заранее предусмотреть:

- хранение секретов отдельно от основной БД, если платформа это позволяет;
- минимизацию логирования чувствительных полей;
- явное разделение пользовательских данных и диагностических дампов;
- осторожное обращение с сырыми email-сообщениями в debug-режиме.


### 14. Что достаточно для MVP

Для первой версии архитектура не должна быть чрезмерно сложной.

Достаточный MVP-набор:

- один desktop-клиент Tauri;
- один frontend;
- один Rust backend;
- SQLite;
- IMAP/SMTP gateway;
- базовый sync engine;
- команды для постов, комментариев и подписок;
- экран синхронизации и диагностики;
- outbox/inbox журнал;
- materialized views для ленты и тредов.

Что можно отложить:

- plugin system;
- split на отдельные процессы;
- полноценную event bus архитектуру внутри клиента;
- сложный offline conflict resolution beyond owner-truth model;
- тонкую систему наблюдаемости;
- многопрофильность и многопользовательский режим.


### 15. С чего начать реализацию

Практический порядок может быть таким:

1. Определить `domain model`.
2. Определить формат `domain event`.
3. Определить формат `protocol message`.
4. Спроектировать SQLite-схему.
5. Реализовать `application commands` для поста и комментария.
6. Реализовать `outbox` и локальную отправку.
7. Реализовать `inbox ingest` и применение событий.
8. Поверх этого строить UI ленты, постов и тредов.

Если пытаться начинать с UI без этих границ, приложение быстро превратится в набор экранов без устойчивой модели данных.
