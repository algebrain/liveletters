## Желаемое разбиение LiveLetters на приложения и библиотеки

### 1. Назначение документа

Этот документ фиксирует желаемую верхнеуровневую структуру проекта LiveLetters как набора отдельных приложений и библиотек.

Он дополняет документ [tauri-client-structure.md](./tauri-client-structure.md), где описана внутренняя структура Tauri-клиента. Здесь акцент сделан не на слоях одного приложения, а на разбиении monorepo на самостоятельные подпроекты.


### 2. Общий принцип

LiveLetters желательно строить не как один большой Tauri-проект, а как monorepo из:

- двух приложений верхнего уровня;
- набора Rust-библиотек;
- набора ClojureScript-библиотек.

Верхнеуровневая идея:

- frontend-app отвечает за пользовательский интерфейс;
- rust-backend-app отвечает за desktop-host, Tauri integration и исполнение backend-логики;
- остальная логика по возможности выносится в переиспользуемые библиотеки с четкими границами ответственности.

Такой подход нужен, чтобы:

- не смешивать UI, доменную модель, транспорт и хранение;
- упростить тестирование;
- сделать архитектуру переиспользуемой;
- упростить развитие системы без постоянного роста связанности;
- подготовить базу для будущих клиентов и вспомогательных инструментов.

При этом подпроекты считаются независимыми:

- у них нет обязательного общего workspace в корне;
- каждый подпроект имеет собственные конфигурационные файлы;
- зависимости между подпроектами должны быть явными.


### 3. Верхнеуровневая структура

Проект должен состоять из двух приложений:

- `liveletters-frontend-app`;
- `liveletters-rust-backend-app`.

И набора библиотек:

- Rust-библиотеки для domain, protocol, storage, sync, mail и application core;
- ClojureScript-библиотеки для UI components, UI model и frontend integration.

Физическая структура репозитория:

- `apps/` для приложений;
- `modules/` для библиотек и прочих модулей.

Рекомендуемый принцип:

- приложения должны быть тонкими;
- основная логика должна жить в библиотеках;
- библиотеки должны иметь ясный API и отдельные тесты;
- границы между библиотеками должны быть важнее удобства "положить все рядом".


### 4. Приложения верхнего уровня

#### 4.1. `liveletters-frontend-app`

Это frontend-приложение на ClojureScript.

Его ответственность:

- маршрутизация;
- экраны и пользовательские сценарии;
- интеграция с backend API;
- отображение ленты, блогов, постов, комментариев и настроек;
- локальное состояние интерфейса;
- пользовательские взаимодействия.

Это приложение не должно содержать:

- доменную бизнес-логику;
- почтовый транспорт;
- правила валидации протокольных сообщений;
- SQL и логику хранения данных.

#### 4.2. `liveletters-rust-backend-app`

Это Tauri-host и backend-приложение на Rust.

Его ответственность:

- запуск desktop-приложения;
- wiring всех Rust-библиотек;
- регистрация Tauri commands и app events;
- управление жизненным циклом приложения;
- запуск фоновых задач;
- интеграция frontend и backend.

Это приложение должно быть максимально тонким. Его основная задача не реализовывать логику, а собирать систему из библиотек.


### 5. Желаемые Rust-библиотеки

#### 5.1. `liveletters-domain`

Основная предметная модель.

Ответственность:

- сущности;
- value objects;
- инварианты;
- типы доменных событий;
- правила видимости;
- базовые доменные политики.

#### 5.2. `liveletters-protocol`

Описание межклиентского протокола.

Ответственность:

- формат протокольного сообщения;
- envelope;
- MIME-представление;
- технические заголовки;
- payload;
- версионирование схемы;
- сериализация и десериализация.

#### 5.3. `liveletters-store`

Доступ к локальному хранилищу.

Ответственность:

- SQLite-схема;
- миграции;
- репозитории;
- inbox/outbox таблицы;
- materialized views;
- транзакции;
- технические индексы синхронизации.

#### 5.4. `liveletters-app-core`

Прикладной orchestration layer.

Ответственность:

- use cases;
- commands;
- queries;
- orchestration между domain, store, protocol, sync и mail;
- application-level ошибки;
- формирование view-oriented read models для backend API.

#### 5.5. `liveletters-mail`

Почтовый транспорт и низкоуровневая работа с email.

Ответственность:

- IMAP;
- SMTP;
- парсинг писем;
- извлечение MIME-частей;
- доставка исходящих сообщений;
- получение входящих сообщений;
- низкоуровневые транспортные адаптеры.

#### 5.6. `liveletters-sync`

Механизм синхронизации и репликации.

Ответственность:

- ingest входящих сообщений;
- распознавание LiveLetters-сообщений;
- извлечение доменных событий;
- валидация;
- дедупликация;
- anti-replay;
- применение доменных событий;
- обработка отложенных зависимостей.

#### 5.7. `liveletters-diagnostics`

Диагностика и техническая наблюдаемость.

Ответственность:

- журналы ошибок;
- журналы входящих и исходящих сообщений;
- технические статусы;
- инспекция невалидных сообщений;
- данные для экрана диагностики.


### 6. Желаемые ClojureScript-библиотеки

#### 6.1. `liveletters-ui-kit`

Базовые компоненты и визуальная система.

Ответственность:

- кнопки;
- формы;
- списки;
- типографика;
- layout primitives;
- повторно используемые UI-компоненты.

#### 6.2. `liveletters-ui-model`

Слой представления и клиентских моделей.

Ответственность:

- преобразование backend DTO в модели для экранов;
- форматирование данных для UI;
- client-side selectors;
- view helpers;
- логика, связанная именно с отображением, а не с доменной моделью.

#### 6.3. `liveletters-frontend-api`

Интеграционный слой frontend <-> backend.

Ответственность:

- вызов Tauri commands;
- подписка на backend events;
- DTO boundary на стороне frontend;
- client adapters;
- обработка ошибок API для UI.


### 7. Принципы разбиения

Разбиение на библиотеки должно подчиняться нескольким правилам:

- библиотека должна иметь одну ясную ответственность;
- библиотека должна иметь маленький, понятный публичный API;
- библиотека должна быть тестируема отдельно;
- библиотека не должна тянуть за собой UI или инфраструктуру без необходимости;
- зависимости должны идти от приложений к библиотекам, а не наоборот;
- thin app, thick libraries должен быть сознательной целью.

Также важно не дробить проект бессмысленно. Новая библиотека должна появляться не потому, что "так красивее", а потому, что у нее есть реальная самостоятельная ответственность.


### 8. Обязательные требования к тестам

Каждое приложение и каждая библиотека должны быть покрыты тестами.

Это обязательное требование, а не желательное.

Минимальные ожидания такие:

- у каждой Rust-библиотеки есть unit tests;
- у библиотек с нетривиальной интеграцией есть integration tests;
- у `liveletters-protocol` есть tests на совместимость форматов;
- у `liveletters-sync` есть tests на out-of-order delivery, duplicate delivery и replay cases;
- у `liveletters-store` есть tests на миграции и корректность materialized state;
- у `liveletters-app-core` есть tests на ключевые use cases;
- у `liveletters-mail` есть tests на parsing и transport edge cases;
- у ClojureScript-библиотек есть tests на UI model, selectors и адаптеры;
- у frontend-app есть tests на ключевые пользовательские сценарии;
- у rust-backend-app есть smoke/integration tests на сборку приложения и связывание основных модулей.

Практический принцип:

- логика тестируется как можно ближе к библиотеке, где она живет;
- приложения тестируют сборку, интеграцию и пользовательские сценарии;
- библиотеки тестируют собственные контракты и инварианты.


### 9. Обязательные требования к документации

Каждое приложение и каждая библиотека должны быть задокументированы.

Это тоже обязательное требование.

Минимальный набор документации для каждого модуля:

- назначение модуля;
- зона ответственности;
- что модуль знает и чего не знает;
- публичный API или точки интеграции;
- ключевые инварианты и ограничения;
- стратегия тестирования;
- примеры типичных сценариев использования.

Для библиотек особенно важно документировать:

- входные и выходные типы;
- гарантии и негарантии;
- формат ошибок;
- версионирование публичного API;
- зависимость от других подпроектов репозитория.

Для приложений особенно важно документировать:

- как они собираются;
- какие библиотеки подключают;
- как запускаются в development-режиме;
- как тестируются;
- какие внешние зависимости нужны.


### 10. Что не стоит делать

Нежелательные направления:

- превращать каждый экран в отдельную библиотеку;
- выносить слишком мелкие доменные подсистемы в отдельные crates без необходимости;
- держать важную бизнес-логику внутри Tauri glue-кода;
- смешивать protocol, domain и mail transport в одном модуле;
- писать библиотеки без тестов;
- писать библиотеки без документации;
- делать приложения "толстыми", а библиотеки формальными и пустыми.


### 11. Практический стартовый набор модулей

Чтобы не переусложнить первый шаг, можно начать с такого стартового набора:

- `liveletters-frontend-app`;
- `liveletters-rust-backend-app`;
- `liveletters-domain`;
- `liveletters-protocol`;
- `liveletters-store`;
- `liveletters-app-core`;
- `liveletters-frontend-api`;
- `liveletters-ui-model`.

После стабилизации первых вертикальных сценариев можно отдельно выделить:

- `liveletters-mail`;
- `liveletters-sync`;
- `liveletters-diagnostics`;
- `liveletters-ui-kit`.

Это позволит не строить слишком крупный монолит, но и не утонуть в чрезмерной фрагментации на старте.


### 12. Целевая архитектурная формула

Желаемая модель проекта:

- два верхнеуровневых приложения;
- приложения живут в `apps/`;
- библиотеки и прочие модули живут в `modules/`;
- основная логика вынесена в библиотеки;
- каждая библиотека и каждое приложение имеют тесты;
- каждая библиотека и каждое приложение имеют собственную документацию;
- приложения остаются тонкими;
- библиотеки остаются осмысленными и переиспользуемыми;
- подпроекты связаны явно, а не через общий workspace.
