# `liveletters-rust-backend-app` INTERFACE

## Назначение

`liveletters-rust-backend-app` это внешний backend-контур desktop-приложения LiveLetters.

Его задача не в том, чтобы хранить основную бизнес-логику, а в том, чтобы дать верхнему слою две удобные точки входа:

- Rust API для локального вызова из кода;
- Tauri-команды и frontend events для ClojureScript frontend.

Именно этот модуль является тонкой границей между frontend и внутренними Rust-библиотеками:

- `liveletters-app-core`
- `liveletters-diagnostics`
- `liveletters-store`

Практически это означает следующее:

- frontend не должен работать напрямую с `Store`, `AppCore` или `DiagnosticsReader`;
- frontend общается только с командами и DTO этого приложения;
- все преобразования из внутренних read model и ошибок в UI-friendly формат происходят здесь.

## Где находится интерфейс

- crate: `liveletters-rust-backend-app`
- библиотечная точка входа: `src/lib.rs`
- бинарная точка входа: `src/main.rs`
- Tauri runtime boundary доступен только при feature `tauri-runtime`

## Что этот модуль предоставляет наружу

У модуля есть два уровня внешнего интерфейса.

### 1. Rust API

Это API для кода на Rust, который хочет поднять backend локально и вызывать его как обычный объект.

Главный тип здесь:

- `BackendApp`

Через него можно:

- открыть backend на in-memory store;
- открыть backend на файловом store по умолчанию;
- создать пост;
- получить ленту;
- получить тред поста;
- получить агрегированный статус синхронизации;
- получить списки проблемных входящих сообщений и проблемных событий.

### 2. Tauri boundary

Это API для frontend.

При включенном `tauri-runtime` приложение регистрирует команды:

- `create_post`
- `get_home_feed`
- `get_post_thread`
- `get_sync_status`
- `list_incoming_failures`
- `list_event_failures`
- `log_frontend_error`

И публикует frontend events:

- `feed-updated`
- `sync-status-changed`
- `frontend-error`

Фактически это и есть backend contract, который должен считать стабильным модуль `liveletters-frontend-api`.

## Основной объект: `BackendApp`

### Зачем нужен `BackendApp`

`BackendApp` это главный фасад backend-слоя.

Он скрывает от вызывающей стороны детали внутреннего wiring:

- как создается `AppCore`;
- как создается `DiagnosticsReader`;
- как открывается `Store`;
- как внутренние ошибки сводятся к единому типу.

Идея в том, что внешний код работает не с набором библиотек, а с одним объектом приложения.

### Как его создавать

Есть три способа создать `BackendApp`.

#### `BackendApp::from_store(store)`

Используется, когда вызывающий код уже сам подготовил `Store` и хочет передать его в backend.

Это полезно:

- в интеграционных тестах;
- в runtime bootstrap-коде;
- там, где нужен явный контроль над хранилищем.

#### `BackendApp::open_in_memory()`

Поднимает backend поверх временного in-memory SQLite store.

Это полезно:

- для тестов;
- для локальных smoke-сценариев;
- для временных запусков без записи на диск.

#### `BackendApp::open_default()`

Поднимает backend поверх store по умолчанию, который использует стандартный путь хранения проекта.

Это основной режим для обычного runtime-сценария приложения.

#### `BackendApp::open_for_home_dir(home_dir)`

Поднимает backend поверх store, который использует явно заданный home-каталог.

Это полезно:

- для интеграционных тестов;
- для локального запуска нескольких независимых экземпляров;
- для runtime-сценариев, где нужно поведение “как будто `HOME` у процесса другой”.

### Что с ним можно делать

#### `create_post(CreatePostRequest)`

Создает пост через `liveletters-app-core`.

Снаружи это выглядит как одна операция, но внутри она запускает полный use case:

- валидацию входных данных;
- создание доменного поста;
- сохранение поста в локальное хранилище;
- постановку соответствующего сообщения в outbox.

Эта операция ничего не возвращает в payload, потому что для текущего frontend boundary считается достаточным факт успешного завершения. После Tauri-вызова frontend получает обновление через события и повторные query-вызовы.

#### `get_home_feed()`

Возвращает текущую материализованную домашнюю ленту.

Это read-only операция. Она не запускает синхронизацию и не меняет состояние, а только поднимает подготовленную read model из `liveletters-app-core` и отдает ее наружу.

#### `get_post_thread(post_id)`

Возвращает один пост вместе с его комментариями.

Эта операция нужна экрану поста или треда обсуждения. Внешний код передает `post_id`, а backend сам достает пост и связанные комментарии из materialized state.

#### `get_sync_status()`

Возвращает агрегированный диагностический статус синхронизации.

Это не сырой лог и не технический дамп, а уже собранное summary для frontend:

- в каком состоянии сейчас контур синхронизации;
- сколько сообщений успешно применено;
- сколько было дубликатов;
- сколько было replay;
- сколько было malformed, invalid, unauthorized;
- сколько событий осталось отложенными;
- сколько сообщений висит в outbox.

Эта операция строится на основе `liveletters-diagnostics`, а не на прямом чтении store с frontend-стороны.

#### `list_incoming_failures()`

Возвращает список проблемных входящих сообщений.

Сюда попадают сообщения, которые не дошли до нормального applied-состояния. Это API нужно экрану диагностики, где пользователь или разработчик хочет понять, какие письма были malformed, invalid, unauthorized или deferred.

#### `list_event_failures()`

Возвращает список проблемных доменных событий.

Это уже не уровень транспортных писем, а уровень событий, извлеченных из технического payload. Это API полезно, когда нужно понять:

- какое именно событие не применилось;
- почему оно не применилось;
- это invalid, replay, unauthorized или другой apply-status.

## Входные типы запросов

### `CreatePostRequest`

Это внутренний Rust request type для метода `BackendApp::create_post`.

Он нужен как компактная форма передачи входных данных в backend use case. Поля означают:

- `post_id`:
  внешний идентификатор создаваемого поста;
- `resource_id`:
  в каком ресурсе публикуется пост;
- `author_id`:
  кто считается автором;
- `created_at`:
  временная метка в unix seconds;
- `body`:
  текст поста.

Этот тип использует ссылки, а не owned-строки, потому что предназначен именно для локального вызова из Rust-кода.

### `CreatePostCommandRequest`

Это сериализуемая форма того же самого запроса для Tauri command boundary.

Ее смысл простой:

- Tauri получает JSON-like payload;
- десериализует его в `CreatePostCommandRequest`;
- затем backend преобразует его во внутренний `CreatePostRequest`.

То есть `CreatePostCommandRequest` нужен не для бизнес-логики, а как transport DTO на границе frontend -> backend.

### `FrontendErrorLogRequest`

Это DTO для команды `log_frontend_error`.

Он нужен, чтобы frontend мог отправить в backend информацию о своей собственной ошибке. Поля используются так:

- `kind`:
  категория ошибки;
- `message`:
  человекочитаемое сообщение;
- `stack`:
  стек вызовов, если есть;
- `source`:
  источник ошибки, например подсистема или экран;
- `location`:
  дополнительная локализация ошибки, например route или участок кода.

Этот тип не участвует в доменной логике. Его роль чисто техническая: сложить полезную frontend-диагностику в runtime log.

## DTO, которые backend отдает наружу

### `PostSummaryDto`

Это короткое описание одного поста для списка или треда.

Наружу передаются:

- `post_id`
- `resource_id`
- `author_id`
- `created_at`
- `body`
- `visibility`
- `hidden`

Смысл DTO в том, что frontend не получает внутренний Rust-тип `PostSummary` напрямую. Он получает JSON-friendly структуру, которую можно безопасно сериализовать и использовать в ClojureScript.

### `CommentSummaryDto`

Это короткое описание одного комментария.

Наружу передаются:

- `comment_id`
- `post_id`
- `parent_comment_id`
- `author_id`
- `created_at`
- `body`
- `visibility`
- `hidden`

Этот DTO нужен экрану треда и любым view model, которые строят дерево комментариев.

### `HomeFeedDto`

Это контейнер для списка постов домашней ленты:

- `posts: Vec<PostSummaryDto>`

Его роль несложная: отдать frontend уже собранный список постов в одной понятной форме.

### `PostThreadDto`

Это контейнер для экрана поста:

- `post: PostSummaryDto`
- `comments: Vec<CommentSummaryDto>`

Именно эта форма сейчас является официальным backend-ответом для команды `get_post_thread`.

### `SyncStatusDto`

Это сводный технический статус синхронизации, уже пригодный для frontend.

Поля означают:

- `status`:
  общее состояние, например `healthy` или `degraded`;
- `applied_messages`:
  сколько сообщений успешно применено;
- `duplicate_messages`:
  сколько входящих сообщений оказались дубликатами;
- `replayed_messages`:
  сколько сообщений были корректны, но представляли replay уже материализованного состояния;
- `unauthorized_messages`:
  сколько сообщений были структурно валидны, но не прошли проверку прав;
- `invalid_messages`:
  сколько сообщений были декодированы, но противоречили правилам консистентности;
- `malformed_messages`:
  сколько сообщений не удалось корректно распознать как валидный LiveLetters message;
- `deferred_events`:
  сколько событий отложено до появления зависимостей;
- `pending_outbox`:
  сколько исходящих сообщений еще находится в outbox.

Этот DTO нужен, чтобы frontend рисовал status panel без знания внутренней структуры `DiagnosticsSnapshot`.

### `IncomingFailureDto`

Это описание проблемного входящего сообщения.

Поля:

- `message_id`:
  идентификатор транспортного сообщения;
- `status`:
  итоговый статус обработки сообщения;
- `preview`:
  сокращенный и санитизированный фрагмент содержимого для диагностики.

Этот DTO нужен экрану, который показывает именно проблемы входящего транспорта.

### `EventFailureDto`

Это описание проблемного доменного события.

Поля:

- `event_id`:
  идентификатор события;
- `event_type`:
  тип события;
- `resource_id`:
  к какому ресурсу относится событие;
- `apply_status`:
  итог применения, например `invalid`, `replay`, `unauthorized`;
- `failure_reason`:
  текстовая причина, если она есть.

Этот DTO нужен для более точной диагностики, чем просто список плохих писем.

## Ошибки и внешний формат ошибок

### `BackendError`

Это внутренний агрегирующий тип ошибки backend-слоя.

Он нужен затем, чтобы код приложения не прокидывал наружу десятки разных библиотечных ошибок. Вместо этого все сводится к одному типу, внутри которого уже различаются источники:

- ошибка application layer;
- ошибка diagnostics layer;
- ошибка store layer.

То есть `BackendError` это не формат для frontend, а внутренний unified error type.

### `CommandErrorDto`

Это уже внешний формат ошибки для Tauri-команд.

У него три поля:

- `code`
- `message`
- `details`

Смысл такой:

- frontend должен разбирать именно `code`;
- `message` можно показать пользователю или положить в лог;
- `details` это опциональная дополнительная нагрузка, например конкретный `post_id` или `comment_id`.

Сейчас backend гарантированно умеет маппить как минимум такие классы ошибок:

- `not_found`
- `validation_error`
- `app_core_error`
- `diagnostics_error`
- `store_error`
- `internal_error`
- `emit_error`

Для frontend это важный контракт: он должен ориентироваться на коды, а не на точный текст `message`.

## Frontend events

### Зачем нужны события

Не все обновления удобно делать только через “вызвал команду -> получил ответ”.

После некоторых backend-действий frontend должен понять, что какие-то данные стали неактуальными и их нужно перечитать. Для этого используются backend events.

### `FrontendEvent`

`FrontendEvent` это маленький тип-обертка над именем и причиной события.

Он нужен, чтобы backend публиковал события единообразно, а не разбрасывал строковые литералы по коду.

Через него сейчас создаются события:

- `feed_updated()`
- `sync_status_changed()`
- `app_booted()`

Метод `name()` возвращает имя канала, в который событие должно быть отправлено.

### Какие события реально важны frontend

#### `feed-updated`

Означает, что домашняя лента могла измениться.

Обычно после него frontend должен:

- инвалидировать локальный кэш ленты;
- заново вызвать `get_home_feed`.

#### `sync-status-changed`

Означает, что мог измениться диагностический статус синхронизации.

Обычно после него frontend должен:

- заново вызвать `get_sync_status`;
- при необходимости перечитать `list_incoming_failures` и `list_event_failures`.

#### `frontend-error`

Это зарезервированное имя события frontend error contour. Сейчас основной видимый путь для ошибок frontend проходит через команду логирования, но имя события уже объявлено как часть boundary.

### Как выглядит payload события

Payload сейчас очень маленький:

```json
{ "reason": "feed-updated" }
```

или

```json
{ "reason": "sync-status-changed" }
```

Идея здесь в том, что событие пока служит скорее сигналом “перечитай данные”, чем полноценной transport-моделью обновления.

## Tauri commands: что делает каждая команда

### `create_post`

Что делает:

- принимает данные нового поста;
- вызывает backend use case создания поста;
- после успеха публикует `feed-updated`;
- после успеха публикует `sync-status-changed`.

Когда использовать:

- когда пользователь создает новый пост из frontend.

Что важно:

- сама команда не возвращает DTO поста;
- обновление UI ожидается через повторный запрос ленты и статуса.

### `get_home_feed`

Что делает:

- возвращает текущую домашнюю ленту как `HomeFeedDto`.

Когда использовать:

- для первоначальной загрузки главного экрана;
- после `feed-updated`;
- после действий, которые могли изменить список постов.

### `get_post_thread`

Что делает:

- по `post_id` возвращает пост и его комментарии как `PostThreadDto`.

Когда использовать:

- при открытии экрана поста;
- после действий, которые могли изменить комментарии или сам пост.

### `get_sync_status`

Что делает:

- возвращает краткий диагностический статус backend-синхронизации.

Когда использовать:

- для экрана диагностики;
- для статусного индикатора в UI;
- после `sync-status-changed`.

### `list_incoming_failures`

Что делает:

- возвращает список проблемных входящих сообщений.

Когда использовать:

- в диагностическом экране;
- при расследовании проблем синхронизации.

### `list_event_failures`

Что делает:

- возвращает список проблемных доменных событий.

Когда использовать:

- если нужен более точный технический разбор, чем список плохих писем;
- для экрана диагностики событийного контура.

### `log_frontend_error`

Что делает:

- принимает DTO ошибки frontend;
- форматирует строку runtime-лога;
- дописывает ее в `frontend-errors.log`.

Когда использовать:

- когда frontend хочет передать backend информацию о своем падении или исключении;
- когда нужно сохранить ошибки JS/UI в общем runtime log.

Что важно:

- эта команда не влияет на доменное состояние;
- это purely technical diagnostics boundary.

## Runtime helpers

При `tauri-runtime` наружу также доступны helper-функции runtime-уровня.

### `build_tauri_app()`

Собирает Tauri application builder с уже зарегистрированными командами и backend state.

Это функция для bootstrap-кода, а не для пользовательских сценариев.

CLI-опция `--home-dir=...` применяется до вызова этой функции на уровне `main.rs`, так что builder уже стартует с нужным effective home.

### `BackendState`

Это обертка для хранения `BackendApp` внутри Tauri state.

Она нужна, чтобы команды могли безопасно доставать backend и выполнять операции через единый runtime container.

### `runtime_log_dir()`

Возвращает `Option<PathBuf>` для debug log directory runtime-уровня.

Важно:

- если debug logging не включен, функция возвращает `None`;
- если debug logging включен, путь по умолчанию вычисляется как `<effective-home>/.liveletters/runtime-logs`;
- это временный диагностический контур, а не обязательная часть обычного runtime.

Отдельно наружу доступен helper:

- `runtime_log_dir_for_home(home_dir)`

Он нужен, когда вызывающему коду надо явно вычислить debug log dir для конкретного home-каталога.

### `runtime_log_line(...)`

Собирает одну форматированную строку лога frontend error.

Это техническая helper-функция, чтобы формат логов оставался единообразным.

## Что считается стабильным контрактом этого приложения

Если смотреть снаружи, то фактически стабильным программным интерфейсом этого приложения сейчас являются:

1. методы `BackendApp`;
2. DTO-запросы и DTO-ответы, которые участвуют в Tauri boundary;
3. набор зарегистрированных Tauri-команд;
4. имена frontend events;
5. формат `CommandErrorDto`.

Текущее дополнение к этому контракту:

6. bootstrap-first команды `get_bootstrap_state`, `get_settings`, `save_settings`;
7. DTO `BootstrapStateDto` и `SettingsDto`.

Именно эти элементы нельзя менять без синхронного изменения:

- `liveletters-frontend-api`;
- frontend app;
- интеграционных тестов boundary-слоя.

## Что этот модуль специально не делает

Важно понимать границы ответственности.

Этот модуль не должен:

- содержать доменную модель;
- напрямую реализовывать persistence-логику;
- сам решать правила синхронизации;
- разрастаться в “толстый backend monolith” поверх библиотек.

Его правильная роль:

- принять внешний запрос;
- вызвать нужный use case;
- перевести результат в внешний DTO;
- перевести внутреннюю ошибку в внешний error DTO;
- при необходимости отправить frontend event.
